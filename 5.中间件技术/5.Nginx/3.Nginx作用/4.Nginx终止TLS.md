在前面的内容中，我们已经介绍了`Nginx`如何提供静态资源服务、实现反向代理以及负载均衡。在实际生产环境中，`Nginx`还有一个非常重要的职责，就是负责处理`HTTPS`连接，也称为`TLS`终止（`TLS Termination`）。

很多初学者认为，浏览器发起`HTTPS`请求后，会一直以加密形式与后端服务通信，如下所示：

```text
浏览器
    │ HTTPS
    ▼
Gin服务
```

实际上，大多数互联网应用并不是这种部署方式。通常情况下，浏览器建立`HTTPS`连接的对象并不是后端业务服务，而是部署在最前面的`Nginx`。浏览器首先与`Nginx`完成`TLS`握手，建立安全连接，随后所有加密请求都会先发送到`Nginx`。`Nginx`完成请求解密后，再以普通`HTTP`请求的形式转发给后端业务服务，其整体架构如下：

```text
浏览器
    │ HTTPS
    ▼
Nginx
    │ HTTP
    ▼
Gin服务
```

这种由`Nginx`负责建立和终止`TLS`连接的方式，就称为`TLS`终止（`TLS Termination`）。

例如，用户访问如下地址：

```http
https://example.com/api/user/info
```

浏览器首先会与`Nginx`完成`TLS`握手，协商出本次连接使用的会话密钥。随后浏览器使用该会话密钥对`HTTP`请求进行加密，并发送给`Nginx`。`Nginx`收到请求后，通过相同的会话密钥完成解密，恢复出原始的`HTTP`请求。

由于`TLS`加密终止在`Nginx`，因此`Nginx`可以读取请求内容，并根据`location`规则通过`proxy_pass`将请求转发给后端`Gin`服务。后端处理完成后返回`HTTP`响应，`Nginx`再使用当前连接的会话密钥对响应进行加密，并发送回浏览器。

整个请求处理过程如下所示：

```text
浏览器
    │
    │ 发起HTTPS请求
    ▼
Nginx
（TLS握手、证书校验、请求解密）
    │
    │ 转发HTTP请求
    ▼
Gin后端服务
（处理业务逻辑）
    │
    │ 返回HTTP响应
    ▼
Nginx
（响应加密）
    │
    │ 返回HTTPS响应
    ▼
浏览器
```

由此可以发现，真正参与`TLS`握手、数字证书验证以及加解密运算的是`Nginx`，后端业务服务对此通常是无感知的。对于`Gin`而言，它接收到的依然只是普通的`HTTP`请求，因此业务代码与是否启用`HTTPS`没有直接关系。

`Nginx`启用`HTTPS`通常只需要配置服务器证书、私钥以及监听`443`端口即可，例如：

```nginx
server {
    listen 443 ssl;
    server_name example.com;

    ssl_certificate     /etc/nginx/ssl/example.pem;
    ssl_certificate_key /etc/nginx/ssl/example.key;

    location /api/ {
        proxy_pass http://127.0.0.1:8080;

        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

其中，`ssl_certificate`用于指定服务器的数字证书文件，通常为`.pem`格式。该证书包含服务器公钥以及域名、有效期、签发机构等信息。在`TLS`握手过程中，`Nginx`会将证书发送给客户端，由客户端验证证书的合法性，从而确认服务器身份。

`ssl_certificate_key`用于指定与数字证书配套的私钥文件。`Nginx`会利用该私钥完成握手阶段的密码学运算，证明自己持有对应私钥，从而完成身份认证。由于私钥直接关系到服务器安全，因此必须妥善保存在服务器本地，避免泄露。

这两个文件通常是在申请数字证书后获得的。首先需要拥有自己的域名，然后向受信任的`CA`申请数字证书。`CA`完成域名所有权验证后，会签发服务器数字证书。证书申请完成后，就会得到服务器证书文件以及与之对应的私钥文件。

可以将数字证书理解为服务器公开的「身份证」，而私钥则是只有服务器持有的「验证凭证」。客户端可以获取数字证书，但无法获得私钥。只有证书与私钥正确匹配时，`TLS`握手才能成功建立安全连接。

当浏览器访问`https://example.com`时，`Nginx`会向浏览器发送数字证书，完成身份认证和密钥协商，并负责后续整个连接的加密与解密工作，而后端服务始终保持普通`HTTP`通信即可。为了保证所有用户都通过安全连接访问网站，生产环境通常还会额外配置一个监听`80`端口的`server`，将所有`HTTP`请求永久重定向到`HTTPS`：

```nginx
server {
    listen 80;
    server_name example.com;

    return 301 https://$host$request_uri;
}
```

例如，用户访问以下网址：

```http
http://example.com/api/user?id=1
```

浏览器收到`301`重定向响应后，会自动再次请求：

```http
https://example.com/api/user?id=1
```

这样可以保证用户始终通过安全连接访问网站，同时也有利于搜索引擎统一收录`HTTPS`地址。

看到这里，可能会产生一个疑问：为什么`Nginx`转发给后端服务时通常使用`HTTP`，而不是`HTTPS`？

原因在于，大多数生产环境中，`Nginx`与后端服务部署在同一台服务器或同一个受信任的内网环境中，通信链路通常不会经过公网，安全风险相对较低。相比继续使用`HTTPS`，采用`HTTP`可以避免后端重复进行`TLS`握手和加解密运算，也无需为每个后端服务单独维护数字证书，从而降低系统复杂度。因此，常见架构会由`Nginx`统一负责`HTTPS`处理，后端服务专注于业务逻辑开发。

当然，这并不意味着`Nginx`与后端之间只能使用`HTTP`。如果两者之间经过公网，或者部署在不同的数据中心、不同云厂商等不可信网络环境中，那么这一段通信同样需要使用`HTTPS`进行保护，如下所示：

```text
浏览器
    │ HTTPS
    ▼
公网Nginx
    │ HTTPS
    ▼
后端服务
```

此时，浏览器与`Nginx`之间会建立第一条`TLS`连接，而`Nginx`又会作为客户端，与后端服务重新建立第二条`TLS`连接。这两条连接相互独立，分别完成各自的握手、密钥协商和数据加密，并不会共享同一套会话密钥。

因此，所谓的`TLS`终止，并不是结束整个请求，而是结束浏览器与`Nginx`之间这一段安全连接。后续是否继续建立新的`TLS`连接，取决于下一段网络是否需要加密保护。正因如此，现代互联网应用通常都会将数字证书管理、`TLS`握手以及加解密运算统一交由`Nginx`处理，而后端服务则专注于业务逻辑，从而在保证通信安全的同时，也降低了系统部署和维护的复杂度。
