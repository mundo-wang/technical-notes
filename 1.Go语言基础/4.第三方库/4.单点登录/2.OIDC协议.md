### 一、`OIDC`要解决的问题

`OIDC`（`OpenID Connect`）是构建在`OAuth 2.0`授权框架之上的身份认证层，由`OpenID Foundation`制定。

在`OAuth 2.0`的笔记中提到过，`OAuth 2.0`解决的核心问题是：资源拥有者如何在不暴露自身凭证的前提下，授权第三方客户端访问其在资源服务器上的特定资源。这套协议从设计之初就是为「授权」而生，而非为「认证」而生。`AccessToken`本身没有标准化的内容格式，对客户端而言只是一个不透明的凭证，唯一的作用是拿着它去资源服务器换数据。

问题在于，早期很多开发者把`OAuth 2.0`错用成了登录方案，思路是用户在`IdP`完成授权后，客户端拿到`AccessToken`，再用它去请求资源服务器的用户信息接口，若请求成功，便认为用户身份验证通过。这种用法存在明显的安全漏洞：

1. `AccessToken`的颁发对象语义不明确，协议本身不保证这个`Token`一定是颁发给当前客户端的，存在「混淆部署」攻击的风险，即恶意客户端`A`可能把自己合法获取的`AccessToken`转交给客户端`B`，让`B`误以为完成了对应用户的身份验证。
2. 不同`IdP`返回的用户信息接口格式各不相同，没有统一的字段定义，客户端需要为每个`IdP`单独适配。
3. `AccessToken`不包含身份验证发生的时间、验证方式等上下文信息，客户端无法判断本次登录是否足够「新鲜」。

`OIDC`正是为了解决这些问题而诞生的。它在`OAuth 2.0`授权码模式的基础上，新增了一个标准化的身份凭证，即`ID Token`，并明确规定了该凭证必须包含哪些字段、如何签名、如何校验。简单来说，`OAuth 2.0`回答的是「这个客户端能不能访问这份资源」，而`OIDC`回答的是「这个用户究竟是谁」。二者结合后，一次标准的`OIDC`登录流程，本质上就是一次走完整套校验规则的`OAuth 2.0`授权码流程，只是额外多换回了一枚`ID Token`。

二、核心角色与核心概念

`OIDC`沿用了`OAuth 2.0`的角色划分，但给出了更贴近认证场景的命名：

- `End User`：终端用户，即`OAuth 2.0`中的资源拥有者。
- `RP`（`Relying Party`，依赖方）：对应`OAuth 2.0`中的客户端应用，也对应单点登录笔记中的`SP`，它信任`OP`给出的身份认证结果，自身不处理用户名密码。
- `OP`（`OpenID Provider`，身份提供方）：对应`OAuth 2.0`中的授权服务器，也对应单点登录笔记中的`IdP`，负责真正完成用户认证并签发`ID Token`。

几个贯穿全文的核心概念：

`ID Token`：`OIDC`新增的核心产物，本质上是一枚`JWT`，由`OP`签发给`RP`，用于证明『此用户已经在`OP`完成认证』。它的内容是结构化、标准化的，不同`OP`返回的字段含义一致，这一点与`OAuth 2.0`里各家自定义的`userinfo`接口完全不同。

`scope`：`OIDC`要求请求时必须携带`openid`这个`scope`，这是触发`OP`签发`ID Token`的标志位。如果只申请`openid`这一个`scope`，能拿到的用户信息非常有限，通常只有`sub`这一个标识字段；如果想拿到邮箱、姓名等更多信息，需要额外申请`profile`、`email`、`address`、`phone`等标准`scope`。

`UserInfo`端点：`RP`拿到`AccessToken`后，可以请求`OP`提供的`UserInfo`端点，换取比`ID Token`里更详细的用户资料，这一步是可选的，很多场景下`ID Token`本身携带的信息已经够用。

`Discovery`文档：`OP`通常会在`{issuer}/.well-known/openid-configuration`这个固定路径下，提供一份`JSON`格式的元数据文档，里面列出了`authorization_endpoint`、`token_endpoint`、`userinfo_endpoint`、`jwks_uri`等关键地址，以及支持的`scope`、签名算法等信息。`RP`只需要知道`issuer`这一个根地址，剩下的所有端点都可以通过这份文档自动发现，不需要手工硬编码。

`JWKS`（`JSON Web Key Set`）：`jwks_uri`指向的就是`OP`的公钥集合，`RP`用这些公钥来校验`ID Token`的签名。`OP`支持公钥轮换，每个公钥都带有`kid`字段作为标识，`ID Token`的`Header`里也会带上对应的`kid`，`RP`据此找到正确的公钥进行校验，这样`OP`轮换密钥时不会导致正在使用中的`Token`全部失效。

三、`ID Token`的结构与关键`Claims`

`ID Token`的物理结构与`Token`笔记中介绍的`JWT`完全一致，由`Header`、`Payload`、`Signature`三段组成，以`.`连接。`OIDC`真正的贡献在于，它对`Payload`里应该出现哪些字段做出了强制性约定，常见`Claims`如下：

- `iss`：签发者，即`OP`的`issuer`地址，`RP`校验时必须确认这个值与自己配置的`issuer`完全一致。
- `sub`：用户在该`OP`下的唯一标识，是`RP`本地建立用户映射关系时应当使用的主键，而不是邮箱或手机号，因为后两者可能被用户修改。
- `aud`：受众，即该`Token`是颁发给哪个`RP`的，对应`RP`的`client_id`，`RP`校验时必须确认自己在这个列表中。
- `exp`：过期时间，`RP`必须校验当前时间是否已超过这个时间点。
- `iat`：签发时间。
- `auth_time`：用户实际完成身份验证的时间，与`iat`可能不同，例如`OP`存在全局会话时，本次`auth_time`可能是很久之前的一次登录。
- `nonce`：`RP`在发起请求时生成并携带的随机值，`OP`会原样写回`ID Token`，`RP`收到后必须核对一致，用于防止重放攻击，这是`OIDC`相对`OAuth 2.0`新增的安全机制，`OAuth 2.0`本身没有这个概念。
- `at_hash`：`AccessToken`的哈希值，用于让`RP`确认收到的`AccessToken`与这枚`ID Token`确实是同一次请求换出来的，防止两者被错配。

四、三种交互模式

`OIDC`定义了三种获取`ID Token`的流程，统称为`Flow`：

1. 授权码模式（`Authorization Code Flow`），对应`response_type=code`，流程与`OAuth 2.0`授权码模式完全一致，区别只是最后从`token_endpoint`换回的响应体里多了一个`id_token`字段。由于`Code`是通过前端跳转传递、`Token`是通过后端接口换取，敏感信息不会经过浏览器地址栏暴露，是目前安全性最高、也最主流的模式，本笔记后续的`Go`代码也基于这一模式展开。
2. 隐式模式（`Implicit Flow`），对应`response_type=id_token`或`id_token token`，`Token`直接通过浏览器重定向的`URL Fragment`返回，不经过后端校验环节。这种模式存在`Token`通过浏览器历史记录、`Referer`等途径泄露的风险，`OAuth 2.0`安全最佳实践与`OIDC`社区均已不推荐继续使用，新项目不应选择这种模式。
3. 混合模式（`Hybrid Flow`），对应`response_type=code id_token`等组合，前端先直接拿到一部分`Token`用于快速渲染界面，同时拿到`Code`供后端换取完整的`Token`集合，使用场景相对小众，常见于需要兼顾前端体验与后端安全校验的场景。

五、两个关键的安全参数：`state`与`nonce`

这两个参数容易被混淆，但职责完全不同：

`state`：在发起授权请求前由`RP`生成，随请求一起发往`OP`，`OP`在回调时原样带回。它的作用是防止`CSRF`攻击，`RP`需要确认收到的回调中的`state`与自己之前发出去的完全一致，否则说明这次回调可能是攻击者伪造的跳转，必须拒绝处理。这一参数在`OAuth 2.0`中就已存在。

`nonce`：`OIDC`新增的参数，专门用于绑定『这次授权请求』与『最终拿到的`ID Token`』。校验逻辑是，`RP`生成`nonce`并随请求发出，`OP`会把这个`nonce`原封不动写入`ID Token`的`Payload`，`RP`在校验`ID Token`时，需要把`Token`里的`nonce`字段与自己本地保存的值进行比对。这样即使攻击者截获了一枚合法的`ID Token`，也无法在另一次会话中重放使用，因为`nonce`对不上。

无论是`state`还是`nonce`，都需要`RP`在发起跳转之前临时保存下来，常见做法是写入`Redis`，并设置较短的`TTL`（例如`5`分钟），跳转回调命中后立即校验并删除，避免被重复使用。

六、`PKCE`：从移动端延伸到所有客户端的强化措施

`PKCE`（`Proof Key for Code Exchange`）最初是为没有能力保管`client_secret`的公开客户端（如移动端`App`、单页应用）设计的，但目前业界趋势是建议所有类型的客户端，包括有后端、能保管密钥的`RP`，都加上这一层防护，因为它能进一步防止授权码被截获后被冒用。

流程上，`RP`在发起授权请求前生成一个随机字符串`code_verifier`，并对其做`SHA256`哈希后再做`Base64UrlEncode`，得到`code_challenge`。授权请求中携带`code_challenge`，`OP`原样记录下来；等到`RP`用`Code`换`Token`时，需要携带原始的`code_verifier`，`OP`重新计算哈希并与之前记录的`code_challenge`比对，一致才放行。即使授权码在传输过程中被截获，攻击者由于不知道`code_verifier`，也无法完成后续的换`Token`步骤。

七、详细流程步骤

以最主流的授权码模式为例，结合`state`、`nonce`、`PKCE`，一次完整的`OIDC`登录流程如下：

1. 用户访问`RP`的受保护页面，`RP`检测本地无有效会话，生成`state`、`nonce`，并生成`PKCE`所需的`code_verifier`与`code_challenge`，将`state`、`nonce`、`code_verifier`关联存入`Redis`并设置短`TTL`，随后将用户重定向至`OP`的`authorization_endpoint`，请求中携带`client_id`、`redirect_uri`、`scope=openid profile email`、`state`、`nonce`、`code_challenge`等参数。
2. `OP`引导用户完成身份验证（如果已存在全局会话则跳过），用户同意授权后，`OP`生成一次性的`Code`，通过浏览器重定向送回`RP`指定的`redirect_uri`，同时带上原样的`state`。
3. `RP`收到回调后，首先校验`state`是否与之前存入`Redis`的值一致，通过后，使用`Code`与对应的`code_verifier`向`OP`的`token_endpoint`发起请求，换取`AccessToken`、`RefreshToken`以及`ID Token`。
4. `RP`收到`ID Token`后，从`jwks_uri`获取`OP`的公钥（通常会做本地缓存），校验签名、`iss`、`aud`、`exp`，并核对`nonce`是否与之前存入`Redis`的值一致，全部通过后，从`ID Token`的`Payload`中解析出`sub`等用户标识字段。
5. `RP`根据`sub`在本地查找或创建用户记录，建立本地会话（可参考单点登录笔记中`SP`建立本地会话的方式），返回受保护资源给用户。后续如果需要调用`OP`的`UserInfo`端点或其他受`AccessToken`保护的资源，按`OAuth 2.0`笔记中的方式携带`AccessToken`即可，过期后使用`RefreshToken`刷新。

九、`Go`代码讲解

`Go`生态中，`OIDC`客户端逻辑通常组合使用两个库：`golang.org/x/oauth2`负责标准`OAuth 2.0`部分（生成授权地址、换取`Token`），`github.com/coreos/go-oidc/v3/oidc`在此基础上补充`OIDC`特有的能力，包括`Discovery`、`ID Token`签名校验。`HTTP`框架使用`Gin`。

先看初始化部分，这一步通常在服务启动时执行一次：

```go
package main

import (
	"context"

	"github.com/coreos/go-oidc/v3/oidc"
	"golang.org/x/oauth2"
)

var (
	provider     *oidc.Provider
	oauth2Config oauth2.Config
	verifier     *oidc.IDTokenVerifier
)

// initOIDC在服务启动时执行一次，通过issuer完成Discovery，自动获取authorization_endpoint、
// token_endpoint、jwks_uri等地址，避免手工硬编码各个端点，OP轮换地址时无需修改代码
func initOIDC(ctx context.Context, issuer, clientID, clientSecret, redirectURL string) error {
	var err error
	provider, err = oidc.NewProvider(ctx, issuer)
	if err != nil {
		return err
	}

	oauth2Config = oauth2.Config{
		ClientID:     clientID,
		ClientSecret: clientSecret,
		RedirectURL:  redirectURL,
		Endpoint:     provider.Endpoint(),
		Scopes:       []string{oidc.ScopeOpenID, "profile", "email"},
	}

	// verifier负责后续校验ID Token的签名、issuer与audience，公钥会通过jwks_uri按需获取并缓存
	verifier = provider.Verifier(&oidc.Config{ClientID: clientID})
	return nil
}
```

`oidc.NewProvider`会自动请求`{issuer}/.well-known/openid-configuration`并解析出标准端点，`provider.Endpoint()`返回的就是`oauth2.Config`需要的`AuthURL`与`TokenURL`。`provider.Verifier`内部会基于`jwks_uri`构建一个带缓存的`KeySet`，校验时按`ID Token`头部的`kid`去匹配对应公钥，无需自己处理密钥轮换逻辑。

登录入口的处理：

```go
package main

import (
	"crypto/rand"
	"encoding/hex"
	"net/http"
	"time"

	"github.com/coreos/go-oidc/v3/oidc"
	"github.com/gin-gonic/gin"
)

// genRandomString生成指定字节长度的随机十六进制字符串，用于state与nonce，
// 字节长度建议不低于16，保证足够的随机性，避免被枚举或碰撞
func genRandomString(n int) (string, error) {
	b := make([]byte, n)
	if _, err := rand.Read(b); err != nil {
		return "", err
	}
	return hex.EncodeToString(b), nil
}

// handleLogin是登录入口，生成本次登录的state与nonce后重定向到OP的授权端点，
// 二者均需要在跳转前临时存入Redis，供回调阶段比对，防止CSRF与Token重放
func handleLogin(c *gin.Context) {
	state, err := genRandomString(16)
	if err != nil {
		c.AbortWithStatus(http.StatusInternalServerError)
		return
	}

	nonce, err := genRandomString(16)
	if err != nil {
		c.AbortWithStatus(http.StatusInternalServerError)
		return
	}

	// 将state与nonce的映射关系写入Redis，TTL设置为5分钟，超时未完成登录则视为失效
	if err := saveLoginContext(c, state, nonce); err != nil {
		c.AbortWithStatus(http.StatusInternalServerError)
		return
	}

	authURL := oauth2Config.AuthCodeURL(state, oidc.Nonce(nonce))
	c.Redirect(http.StatusFound, authURL)
}

// saveLoginContext以state为Key，将nonce写入Redis并设置5分钟过期时间
func saveLoginContext(c *gin.Context, state, nonce string) error {
	key := "oidc:state:" + state
	return rdb.Set(c, key, nonce, 5*time.Minute).Err()
}
```

这里`oidc.Nonce(nonce)`是一个`oauth2.AuthCodeOption`，它的作用是把`nonce`参数拼接到最终生成的授权地址中，`OP`收到后会原样写入`ID Token`。如果项目同时要求`PKCE`，可以再叠加`oauth2.GenerateVerifier()`与`oauth2.S256ChallengeOption(verifier)`，把`code_verifier`也一并存入`Redis`，换`Token`时通过`oauth2.VerifierOption(verifier)`带上。

回调部分是整个流程里校验最密集的一环：

```go
package main

import (
	"context"
	"net/http"

	"github.com/gin-gonic/gin"
)

// handleCallback处理OP的回调请求，依次完成state校验、code换token、ID Token签名与
// claims校验，任意一步失败都必须中断流程，不能跳过校验直接信任返回的用户信息
func handleCallback(c *gin.Context) {
	ctx := context.Background()

	// 校验回调携带的state是否与登录时写入Redis的值一致，命中后立即从Redis删除，防止重复使用
	nonce, ok := checkAndConsumeState(c, c.Query("state"))
	if !ok {
		c.AbortWithStatus(http.StatusBadRequest)
		return
	}

	// 用授权码向token_endpoint换取AccessToken、RefreshToken与ID Token
	token, err := oauth2Config.Exchange(ctx, c.Query("code"))
	if err != nil {
		c.AbortWithStatus(http.StatusUnauthorized)
		return
	}

	rawIDToken, ok := token.Extra("id_token").(string)
	if !ok {
		c.AbortWithStatus(http.StatusUnauthorized)
		return
	}

	// 校验ID Token的签名、issuer、audience与过期时间，全部由verifier内部完成
	idToken, err := verifier.Verify(ctx, rawIDToken)
	if err != nil {
		c.AbortWithStatus(http.StatusUnauthorized)
		return
	}

	if idToken.Nonce != nonce {
		c.AbortWithStatus(http.StatusUnauthorized)
		return
	}

	var claims struct {
		Sub           string `json:"sub"`
		Email         string `json:"email"`
		EmailVerified bool   `json:"email_verified"`
		Name          string `json:"name"`
	}
	if err := idToken.Claims(&claims); err != nil {
		c.AbortWithStatus(http.StatusInternalServerError)
		return
	}

	// 以claims.Sub作为OP侧用户的唯一标识，查找或创建本地用户绑定关系并落库
	userID, err := upsertLocalUser(claims.Sub, claims.Email, claims.Name, rawIDToken, token)
	if err != nil {
		c.AbortWithStatus(http.StatusInternalServerError)
		return
	}

	c.JSON(http.StatusOK, gin.H{"user_id": userID})
}

// checkAndConsumeState从Redis中取出state对应的nonce，命中后立即删除，避免同一state被重复消费
func checkAndConsumeState(c *gin.Context, state string) (string, bool) {
	key := "oidc:state:" + state
	nonce, err := rdb.Get(c, key).Result()
	if err != nil {
		return "", false
	}
	rdb.Del(c, key)
	return nonce, true
}
```

`token.Extra("id_token")`这一步需要特别说明，`golang.org/x/oauth2`的`Exchange`方法返回的`*oauth2.Token`结构体本身只定义了`AccessToken`、`RefreshToken`、`Expiry`这些`OAuth 2.0`原生字段，`id_token`属于`OIDC`额外塞进响应体的字段，`oauth2`库会把所有未识别的额外字段都放进`Extra`这个`map`里，所以需要用`token.Extra("id_token")`取出来，再做一次类型断言转成字符串。

`upsertLocalUser`负责把`OP`侧的身份与本地用户体系关联起来，结合`Gorm`实现大致是先按`sub`查询`oidc_identity`表，不存在则连同本地`user`表一并创建，存在则更新最新的`Token`信息，核心原则是必须以`sub`作为关联键，而不是`email`，因为`email`在多数`OP`上是允许用户修改的。

为了持久化这层绑定关系与`Token`信息，可以设计如下表结构：

```sql
CREATE TABLE `oidc_identity` (
    `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    `user_id` BIGINT NOT NULL COMMENT '本地用户ID',
    `op_issuer` VARCHAR(255) NOT NULL COMMENT 'OP的issuer标识',
    `subject` VARCHAR(255) NOT NULL COMMENT 'OP侧用户唯一标识sub',
    `access_token` VARCHAR(1024) NOT NULL DEFAULT '' COMMENT 'OP签发的AccessToken',
    `refresh_token` VARCHAR(1024) NOT NULL DEFAULT '' COMMENT 'OP签发的RefreshToken',
    `id_token` VARCHAR(2048) NOT NULL DEFAULT '' COMMENT '最近一次签发的ID Token原文',
    `at_expired_at` DATETIME NOT NULL COMMENT 'AccessToken过期时间',
    `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    PRIMARY KEY (`id`)
) COMMENT='OIDC身份绑定表';
```

这里没有把`subject`设置成唯一索引，是因为按表结构规范，除主键外不额外建索引，实际业务中按`op_issuer`加`subject`查询的场景非常频繁，如果数据量上来后查询变慢，可以再单独评估补充索引。

十、`OIDC`与`OAuth 2.0`、`SAML`、`CAS`的关系

回到最初的问题，`OIDC`并不是`OAuth 2.0`的替代品，而是寄生在`OAuth 2.0`授权码模式之上的一层身份语义补充，二者的端点、授权码、`Token`换取方式完全复用，区别仅在于`OIDC`多定义了`ID Token`这一标准化产物，并新增了`nonce`这样的防重放校验。

与单点登录笔记中提到的`SAML 2.0`、`CAS`相比，三者解决的都是跨系统统一认证的问题，但所处的时代与生态不同：`SAML 2.0`基于`XML`，协议厚重，多见于企业级集成；`CAS`票据机制简单，常见于高校与早期系统；`OIDC`构建在`OAuth 2.0`之上，使用`JSON`与`JWT`，更贴合现代互联网应用与移动端场景，目前互联网公司新做的`SSO`系统，绝大多数都会选择`OIDC`作为底层协议。
