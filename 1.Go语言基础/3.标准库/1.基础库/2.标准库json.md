在本节中，我们将讨论如何在结构体对象或`map[string]interface{}`与`JSON`字符串之间进行相互转换。这里的结构体对象可以是单个结构体实例，也可以是结构体切片。

### 1. 结构体对象和`map[string]interface{}`转换成`JSON`字符串（序列化）

我们使用`json.Marshal`这个函数，函数签名如下：

```go
func Marshal(v interface{}) ([]byte, error)
```

这里的参数`v`可以是结构体对象、映射、切片本身，或它们的引用，得到的是`[]byte`类型的`JSON`格式字节数组，这个过程被称为序列化。得到`[]byte`类型对象后，再通过内置函数`string()`将其转换为`JSON`字符串。

代码示例如下所示：

```go
type Person struct {
	Name string `json:"name"`
	Age  int    `json:"age,omitempty"`
}

func main() {
	person := Person{
		Name: "Alice",
		Age:  30,
	}
	personJsonByte, _ := json.Marshal(person)
	personJsonStr := string(personJsonByte)

	dataMap := map[string]interface{}{
		"name": "Bob",
		"age":  25,
	}
	mapJsonByte, _ := json.Marshal(dataMap)
	mapJsonStr := string(mapJsonByte)

	personSlice := []Person{
		{Name: "Alice", Age: 30},
		{Name: "Bob", Age: 25},
	}
	personSliceJsonByte, _ := json.Marshal(personSlice)
	personSliceJsonStr := string(personSliceJsonByte)
}
```

> **注意：**结构体中的私有字段在序列化时不会被处理，即便这些字段定义了`json`标签。

如果输入参数`v`为`nil`，调用`Marshal`函数时，不会返回`error`，而是返回字节切片`[110, 117, 108, 108]`，对应的`ASCII`字符为`null`。使用`string()`函数转换后，结果将是字符串`"null"`。

在对结构体对象进行序列化时，若某个字段未被赋值（零值），默认情况下该字段仍会出现在生成的`JSON`中。此时，可以在结构体的可选字段上使用`json`标签中的`omitempty`选项，使该字段在为零值时不会出现在`JSON`中，从而避免输出冗余字段。

但需要注意的是，如果某字段的零值本身具有特定业务含义，则不应使用`omitempty`，以免导致数据语义丢失。如果希望完全从序列化结果中排除某个字段，可以使用`json:"-"`标签。使用`json:"-"`与不添加`json`标签的区别在于，前者会使该字段在输出`JSON`字符串时被忽略，而后者则会使用结构体字段名（即大写字母开头的）作为`JSON`的参数名。

如果输入参数`v`是一个空结构体或所有字段都被忽略的结构体对象，使用`Marshal`函数时，将返回字节切片`[123, 125]`，对应的`ASCII`字符为`{}`。通过`string()`函数转换后，结果将是字符串`"{}"`。

在结构体`json`标签中加上`,string`选项，可在序列化时，将整型、浮点型和布尔类型字段以字符串形式输出。例如：

```go
type Example struct {
	ID     int     `json:"id,string"`
	Amount float64 `json:"amount,string"`
	Valid  bool    `json:"valid,string"`
}
```

上面的结构体对象序列化后的`JSON`格式如下：

```json
{
  "id": "123",
  "amount": "456.78",
  "valid": "true"
}
```

例如，雪花算法生成的`ID`字段通常是`19`位的十进制数字。当该字段以`JSON`格式传递给前端时，如果以整型传输，由于`JavaScript`的`Number`类型存在精度限制，无法精确表示超过`2^53 - 1`的整数，会导致出现精度丢失的问题。为避免这一情况，可通过在结构体字段上添加`,string`标签，将该字段序列化为字符串形式进行传输，从而确保数据的准确性。

目前，`encoding/json`标准库只支持`,string`这一种类型转换标签，用于将基础类型序列化为`JSON`字符串格式。

对于通过`json.Marshal`函数序列化得到的`JSON`数据，在转换为`string`类型并打印时，其输出格式默认是紧凑的平铺形式，不包含换行或额外的空白字符，打印效果如下所示：

```json
{"name":"Alice","age":30}
```

如果我们想将`JSON`字符串进行格式化输出，可以使用`json.MarshalIndent`方法，如下所示：

```go
formatPersonJson, _ := json.MarshalIndent(person, "", "  ")
fmt.Println(string(formatPersonJson))
```

这样打印出来的字符串效果如下所示：

```json
{
  "name": "Alice",
  "age": 30
}
```

`MarshalIndent`函数的第二个参数用于指定每一行的前缀，通常传入空字符串；第三个参数用于指定缩进字符，一般使用两个空格或一个制表符。无论`JSON`结构中包含多少层嵌套或数组元素，该函数都能够按照规范的格式进行输出。

### 2. `JSON`字符串转换成结构体对象和`map[string]interface{}`（反序列化）

使用到`json.Unmarshal`函数，函数签名如下：

```go
func Unmarshal(data []byte, v interface{}) error
```

第一个参数是`[]byte`类型的`JSON`，第二个参数必须是结构体对象、切片或`map`的引用。这个过程被称为反序列化。

代码示例如下所示：

```go
type Person struct {
	Name string `json:"name"`
	Age  int    `json:"age"`
}

func main() {
	jsonStr := `{"name":"Alice","age":30}`
    
	var person Person
	_ = json.Unmarshal([]byte(jsonStr), &person)
    
	var personMap map[string]interface{}
	_ = json.Unmarshal([]byte(jsonStr), &personMap)
    
    jsonSliceStr := `[
		{"name":"Alice","age":30},
		{"name":"Bob","age":25}
	]`
    
    var personSlice []Person
    _ = json.Unmarshal([]byte(jsonSliceStr), &personSlice)
}
```

> 在`JSON`规范中，空格、制表符和换行符都属于可忽略的空白字符。`encoding/json`包中的`Unmarshal`函数在解析`JSON`数据时，会自动跳过这些空白字符，仅关注结构符号以及具体的字段内容。

在使用`Unmarshal`函数时，需要确保入参`data`既不为`nil`，也不是空字节数组，否则会返回错误`unexpected end of JSON input`。同时，入参`v`必须是非`nil`的指针，这一点非常容易被忽视，实践中经常踩坑，例如下面代码：

```go
jsonStr := `{"name":"Alice","age":30}`
var person *Person
err := json.Unmarshal([]byte(jsonStr), person)
```

在这段代码中，`person`的类型虽然是`*Person`，但在初始化时其值为`nil`。由于目标指针未指向有效的内存对象，执行反序列化时会失败，并返回错误：`json: Unmarshal(nil *Person)`。但是在上述代码中，如果最后一行代码这样写：

```go
err := json.Unmarshal([]byte(jsonStr), &person)
```

可以发现，反序列化依然能够成功。原因在于，传递给`Unmarshal`的变量类型是`**Person`。在`encoding/json`的反序列化过程中，有一条关键规则：当目标类型是指向指针的指针，且该指针当前为`nil`时，`Unmarshal`会自动为其分配内存，然后将解析得到的结果写入新分配的指针对象中，最后再将该指针对象的地址赋值给外层指针。

将字节数组反序列化成结构体对象时，给结构体字段添加`json`标签中的`omitempty`选项没有任何意义。

在进行`JSON`反序列化时，如果结构体字段未显式声明`json`标签，匹配`JSON`键名时仅会执行大小写不敏感匹配。例如字段名为`Name`时，`Name`、`name`或`nAme`都可以成功映射到该字段。需要注意的是，这种匹配机制只处理大小写差异，并不会进行命名风格转换。当结构体字段名为`UserName`时，`JSON`中的`user_name`由于包含下划线，不属于大小写差异范畴，因此无法自动匹配。

在进行反序列化时，`json.Unmarshal`并不要求目标结构体对象是零值对象。`JSON`的反序列化过程仅会将`JSON`中存在的字段值覆盖到结构体对象中对应的字段。如果结构体中包含`JSON`中没有的字段，目标结构体对象中该字段的值将保持不变。如果结构体在反序列化前是零值对象，而`JSON`中没有该字段，那么该字段最终就是该类型的零值。

如果`JSON`中有某个字段在`struct`里没定义，`Go`反序列化时不会报错，它会忽略这个字段。如下所示：

```go
jsonStr := `{"name":"Alice","age":30,"address":"蔡徐村"}`
var person Person
_ = json.Unmarshal([]byte(jsonStr), &person)
```

结构体对象`person`会正常反序列化`JSON`字符串的`name`和`age`字段，而忽略结构体中不存在的`address`字段。

如果某个字段的结构不固定，或希望先完成外层反序列化、再按需解析内层数据，可以将该字段类型声明为`json.RawMessage`。

`json.RawMessage`的底层类型是`[]byte`，但它实现了`json.Marshaler`和`json.Unmarshaler`接口，序列化时会原样输出，反序列化时也会原样写入，不会被二次转义。例如下面的结构体：

```go
type Response struct {
    Code    int              `json:"code"`
    Message string           `json:"message"`
    Data    json.RawMessage  `json:"data"`
}
```

以上面的结构体为例，`Data`字段的实际内容可能因业务场景而异，拿到外层的`Code`和`Message`后，再按需决定如何解析`Data`：

```go
jsonStr := `{"code":0,"message":"ok","data":{"name":"Alice","age":30}}`
var resp Response
_ = json.Unmarshal([]byte(jsonStr), &resp)

// resp.Data此时为原始JSON字节，后续按需解析
var person Person
_ = json.Unmarshal(resp.Data, &person)
```

在进行反序列化时，必须确保字段类型之间具有兼容性，否则将返回类型无法兼容的`error`。对于`JSON`中的`string`类型字段，即使结构体中对应字段的类型不是`string`，只要在字段标签中添加`,string`选项，仍然可以完成反序列化。

### 3. 时间格式的处理

首先需要明确，时间格式（例如`time.Time`，或`Java`中的`LocalDateTime`）在经过`JSON`序列化后，会以字符串的形式进行表示。

`Java`的`Jackson`库提供了一个注解`@JsonFormat`，用于在将`Java`对象序列化为`JSON`字符串，或将`JSON`字符串反序列化为`Java`对象时，对特定字段的格式进行自定义处理。该注解特别适用于处理日期和时间类型字段：

```java
@JsonFormat(pattern = "yyyy-MM-dd HH:mm:ss", timezone = "Asia/Shanghai")
private LocalDateTime createTime;
```

`Go`语言并未提供类似的格式指定方法。`encoding/json`在序列化时，会根据字段类型和`tag`自动转换为固定格式。`time.Time`类型字段默认会被序列化为`2006-01-02T15:04:05.999999Z07:00`格式，例如`2025-07-07T17:25:31.239132+08:00`。

这是因为`Java`具有运行时反射和注解处理能力，`Jackson`能够通过扫描注解（如`@JsonFormat`）动态决定序列化行为。而`Go`虽然支持反射机制，但没有注解语法，因此只能通过`struct`的`tag`来控制序列化行为，功能相对有限。

前端向后端传递时间数据时，应采用字符串形式的`RFC3339`格式，例如`2006-01-02T15:04:05Z07:00`。后端在反序列化时，会将该字符串解析如果`JSON`中有某个字段在`struct`里没定义为`time.Time`类型。需要注意的是，默认情况下不支持解析`2006-01-02 15:04:05`这种格式的时间字符串。

这体现了`Go`语言的「大道至简」哲学。时间的序列化形式并不重要，关键在于能否正确地反序列化。虽然`RFC3339`格式对人类来说不直观，但它可以被`Go`的`time.Parse()`直接解析，也能被`JavaScript`、`Python`等大多数编程语言兼容解析，同时可保留微秒精度和时区等完整信息。一旦对象被序列化为`byte[]`，其「人类可读性」就不再是关注点，重点转向了机器可读性。

### 4. 接口字段反序列化类型丢失问题

`JSON`的序列化过程仅包含值信息，而不携带类型信息，但反序列化阶段却依赖类型信息，这会引发类型丢失问题。

以如下的接口与结构体的定义为例：

```go
type Animal interface {
	Speak()
}

type Dog struct {
	Name string `json:"name"`
}

func (d *Dog) Speak() {
	fmt.Println("Wang!")
}

type Cat struct {
	Name string `json:"name"`
}

func (c *Cat) Speak() {
	fmt.Println("Miao~")
}

type Zoo struct {
	Animal Animal `json:"animal"`
}
```

我们构造一个`Zoo`对象并进行序列化：

```go
zoo := Zoo{
	Animal: &Dog{Name: "Wangcai"},
}
zooByte, _ := json.Marshal(zoo)
```

随后执行反序列化，并输出`Animal`字段的实际类型：

```go
var zoo2 Zoo
_ = json.Unmarshal(zooByte, &zoo2)
fmt.Printf("%T\n", zoo2.Animal)  // <nil>
fmt.Printf("%T\n", &zoo2.Animal) // *main.Animal
```

可以看到具体类型已经丢失。这是因为`JSON`中只包含`{"name":"Wangcai"}`，`Go`无法判断应实例化为`Dog`、`Cat`还是其他实现该接口的类型。`encoding/json`库的默认行为是：当反序列化目标是接口且未预先指定具体对象时，它无法确定应填充的实际类型，对于未指向任何具体实现的接口变量，其默认值为`nil`。

这会直接导致下面的类型断言失败：

```go
if dog, ok := zoo2.Animal.(*Dog); ok {
    // 对nil类型做类型断言，ok值永远为false
}
```

而在`Go`中，类型断言只能在接口值上进行，而不能在指向接口的指针上进行。所以下面代码会有编译错误：

```go
animal := &zoo2.Animal
// 会发生编译错误
if dog, ok := animal.(*Dog); ok {
	
}
```

这将直接破坏基于接口的多态机制，使`Dog`类型拥有的字段、所实现的方法永久失效。

如果不关注序列化结果的可读性，可以考虑使用`gob`替代`json`。`encoding/gob`是为`Go`类型系统设计的二进制序列化协议，会在编码过程中携带类型描述信息（`type descriptor`），从而在反序列化时能够还原具体类型。

下面是使用`gob`进行序列化与反序列化的完整示例：

```go
// 需要将所有可能赋值给接口的具体类型进行注册
gob.Register(new(Dog))
gob.Register(new(Cat))

zoo := Zoo{
	Animal: &Dog{Name: "Wangcai"},
}

var buf bytes.Buffer
encoder := gob.NewEncoder(&buf)
_ = encoder.Encode(zoo)

var zoo2 Zoo
decoder := gob.NewDecoder(&buf)
_ = decoder.Decode(&zoo2)
fmt.Printf("%T\n", zoo2.Animal) // *main.Dog
if dog, ok := zoo2.Animal.(*Dog); ok {
	fmt.Println("Dog name:", dog.Name) // 类型断言成功
}
```

使用`gob`必须显式注册具体类型，否则，当`gob`处理接口字段时无法识别其实际类型，`Encode`会直接返回错误：

```sh
gob: type not registered for interface: *main.Dog
```

其底层机制是：`gob`会在编码流中写入类型定义（类似类型表），后续数据通过引用这些定义进行编码，因此在反序列化阶段可以正确完成`interface{}`到具体类型（如`Dog`）的还原。

需要注意，若`Dog`结构体中也存在接口字段，则所有可能赋值给该接口字段的具体类型同样需要提前注册。

在实际工程中，更常见的方案不是直接替换为`gob`，而是在`JSON`结构中显式引入类型标识字段，例如：

```json
{
  "type": "dog",
  "animal": {
    "Name": "Wangcai"
  }
}
```

此时，`JSON`不仅用于数据传输，还承担了类型分发的职责。

这就要求我们实现序列化方法，格式如下所示：

```go
func (t T) MarshalJSON() ([]byte, error)
```

方法名固定为`MarshalJSON`，接收者通常为值类型，方法无入参，返回的`[]byte`表示最终生成的`JSON`数据，开发者可以在方法内部自行构造输出结构，例如根据实际类型写入`type`字段，从而实现多态结构的编码。

随后，需要在被序列化的类型上实现自定义解码方法，格式如下所示：

```go
func (t *T) UnmarshalJSON(data []byte) error
```

方法名固定为`UnmarshalJSON`，接收者必须为指针类型，`[]byte`类型的入参`data`表示原始`JSON`字节流，用于实现自定义反序列化逻辑，例如根据`type`字段将数据解析为不同类型结构体。

这样一来，无需修改`Zoo`结构体定义，也能够实现多态的序列化与反序列化行为。随后在使用`json.Marshal`与`json.Unmarshal`对该结构体对象进行处理时，便会自动调用我们自定义的`MarshalJSON`与`UnmarshalJSON`方法。
