在`Go`语言中，测试函数的编写简洁直观。与`Java`不同，`Go`的测试文件通常与被测试文件位于同一目录，且文件名必须以`_test.go`结尾。这是一项硬性规定，未遵循此命名规则的文件不会被`Go`编译器识别为测试文件，其中的测试函数也无法执行。

若被测试文件名为`math.go`，则对应的测试文件应命名为`math_test.go`。这种命名方式便于开发者快速定位源文件与其测试文件之间的对应关系。值得一提的是，在`GoLand`中，测试文件通常以浅绿色背景显示，进一步方便开发者加以区分：

编写`Go`语言的测试函数时，需要遵循以下规则：

1. 测试函数名必须以`Test`开头，并遵循驼峰命名法，否则`Go`编译器不会将其识别为测试函数，导致无法执行。
2. 测试函数有且仅有一个参数，即`t *testing.T`，这是`Go`语言内置的测试类型参数。
3. 测试函数的命名应与被测试函数保持对应。例如被测试函数名为`AddNums`，则对应的测试函数应命名为`TestAddNums`。
4. 被测试方法对应的测试函数命名规则与此保持一致。例如结构体名称为`Users`、方法名称为`Query`，则对应的测试函数应命名为`TestUsers_Query`。

例如被测试文件的代码如下所示：

```go
func AddNums() {
	// some logic
}

type Users struct {
	// some field
}

func (u *Users) Query() {
	// some logic
}
```

它对应的测试文件代码如下：
```go
import (
	"fmt"
	"testing"
)

func TestAddNums(t *testing.T) {
	// some logic
}

func TestUsers_Query(t *testing.T) {
	// some logic
}
```

点击测试函数左侧的绿色小箭头，就可以对这个测试函数进行单独运行：

接下来介绍`t *testing.T`对象。`*testing.T`是一个指向测试状态的结构体指针，封装了测试过程中所需的状态与功能，可用于报告测试失败、记录日志等操作。以下是该类型的几个常用方法。

#### 1. `Run`方法：

在一个测试函数中，可以使用`Run`方法运行多个子测试以覆盖不同的测试场景。子测试失败时不会影响其他子测试的执行，且每个子测试都会单独报告其结果。它的方法签名如下：

```go
func (t *T) Run(name string, f func(t *T)) bool
```

下面是其使用示例：

```go
import (
    "testing"
)

func TestExample(t *testing.T) {
    t.Run("Subtest1", func(t *testing.T) {
        // 这里是子测试1的逻辑
        // 使用t.Error或者t.Fail来标记测试失败
    })
    t.Run("Subtest2", func(t *testing.T) {
        // 这里是子测试2的逻辑
        // 使用t.Error或者t.Fail来标记测试失败
    })
}
```

这里我们看到，每个子测试左侧都有一个绿色小按钮，我们可以单独运行一个子测试，也可以运行整个测试：

#### 2. `Error`和`Errorf`方法

用于报告测试失败的情况。使用示例如下：

```go
t.Error("测试失败")
t.Errorf("测试失败：%v", err)
```

#### 3. `Fatal`和`Fatalf`方法

用于报告测试失败并立即终止当前测试函数的执行，适用于前置条件不满足、继续执行会导致`panic`或产生无意义结果的场景。

使用示例如下：

```go
t.Fatal("致命错误，终止测试")
t.Fatalf("致命错误：%v", err)
```

#### 4. `Skip`和`Skipf`方法

用于跳过当前测试，测试结果记为`SKIP`而非`FAIL`，常用于某些依赖环境或外部条件不满足时的条件跳过。使用示例如下：

```go
t.Skip("跳过该测试")
t.Skipf("跳过该测试：%s", reason)
```

#### 5. `Log`和`Logf`方法

用于记录执行时的日志信息。使用示例如下：

```go
t.Log("这是一个日志信息")
t.Logf("这是一个格式化的日志信息：%s", msg)
```

其余的方法，需要使用的时候再进行查阅即可。

接下来介绍`go test`命令。`go test`是执行测试的核心命令，上面提到的绿色箭头本质上是`GoLand`对`go test`系列命令的图形化封装，点击时底层实际执行的是`go test`加上`-run`参数来定位具体的测试函数，命令如下：

```sh
# 只运行TestAddNums
go test -run TestAddNums .
# 只运行TestExample中名为Subtest1的子测试
go test -run TestExample/Subtest1 .
```

执行`go test .`命令时，`Go`会编译当前目录对应的包，并额外编译其中的`_test.go`文件，生成一个临时二进制文件。随后`Go`会执行该文件，由其中的测试运行器自动调用所有符合命名规则的测试函数。测试执行完成后，默认会自动删除该临时二进制文件。

其中`.`表示当前目录，也可以替换为其他任意路径。使用`go test ./...`可以递归测试当前模块下的所有包，此时`Go`会为每个包分别编译并生成独立的测试二进制文件，这些二进制文件会在测试执行完成后自动删除。

测试函数的执行顺序通常受测试文件处理顺序和函数声明顺序影响。不同`_test.go`文件通常按文件名字典序处理，同一文件内的测试函数通常按声明顺序执行。例如，包内存在`a_test.go`和`b_test.go`两个测试文件时，通常会先执行`a_test.go`中的测试函数，再执行`b_test.go`中的测试函数；同一文件中定义的多个测试函数，通常按照从上到下的顺序依次执行。

需要注意的是，这种执行顺序不属于语言规范保证的内容，因此不应作为测试之间建立依赖关系的依据。每个测试函数都应保持相互独立，不能假设某个测试一定先于另一个测试执行，否则在调整文件名称、重构代码或升级`Go`版本后，都可能导致测试结果变得不稳定甚至出现偶发失败。如果需要表达明确的执行关系，建议通过子测试`t.Run()`在同一个测试函数内组织测试流程。

若希望某个测试函数与其他测试函数并行执行，需要在函数体开头显式调用`t.Parallel()`，此时`Go`会将其纳入并行调度。子测试同理，默认串行，调用`t.Parallel()`后会与其他子测试并行运行。代码如下所示：

```go
func TestFoo(t *testing.T) {
    t.Parallel() // 标记该测试函数参与并行执行
    // some logic
}

func TestBar(t *testing.T) {
    t.Parallel()
    t.Run("Subtest1", func(t *testing.T) {
        t.Parallel() // 子测试同样需要单独声明才能并行
        // some logic
    })
    t.Run("Subtest2", func(t *testing.T) {
        t.Parallel()
        // some logic
    })
}
```

若需保留编译产物以供调试，可使用`-c`参数，此时不会执行测试，而是将测试二进制文件保存到当前目录：

```sh
# 编译但不执行，将测试二进制保存为「包名.test」
go test -c .

# 通过-o参数自定义输出文件名
go test -c -o math.test .
```

需要注意的是，`-c`参数存在一个限制：它只能作用于单个包，无法与`./...`组合使用。若需保留所有包的测试二进制，只能逐包手动执行，或通过脚本遍历各包分别调用`go test -c`命令。

当测试包中某个文件定义了`TestMain`函数时，`go test`会将测试流程的控制权交给`TestMain`，由它负责在合适的时机调用方法执行各项测试，并可在前后插入自定义逻辑。其函数签名如下：

```go
func TestMain(m *testing.M)
```

`m *testing.M`是测试管理对象，调用`m.Run()`会触发包内所有测试函数的执行，返回值为退出码。必须将该退出码传递给`os.Exit`，否则测试结果将无法正确反映到`CI/CD`流水线或命令行的退出状态中。

使用示例如下所示：

```go
import (
    "fmt"
    "os"
    "testing"
)

func TestMain(m *testing.M) {
    fmt.Println("测试开始前：初始化资源")
    // 在这里进行数据库连接、配置加载等初始化操作
    code := m.Run() // 执行包内所有测试函数
    fmt.Println("测试结束后：清理资源")
    // 在这里进行资源释放、数据清理等操作
    os.Exit(code)
}
```

需要注意，`TestMain`的作用域是整个测试包，而非单个测试文件。一个包内只能定义一个`TestMain`，定义多个会导致编译报错。
