在默认存储引擎`InnoDB`下，`MySQL`为了保证数据不丢失，主要依赖于以下几个机制：

#### 1. 预写日志机制（`WAL`）

在`MySQL`的`InnoDB`存储引擎中，预写式日志（`Write-Ahead Logging`，`WAL`）机制确保了数据的持久性和一致性。具体流程如下：

当执行`INSERT`或`UPDATE`语句时，`InnoDB`会首先将变更操作记录到`Redo Log`中。这个日志会被写入磁盘，确保即使发生崩溃或断电等故障，数据也能恢复。只有在`Redo Log`成功写入磁盘后，`InnoDB`才会向客户端返回操作成功的响应。

随后，后台线程会定期将内存中的数据页刷新到磁盘上的数据文件中，这个过程称为写回（`Flush`）。数据页的刷新并非立即进行，而是基于一定的策略（如日志写满、缓冲池满或时间间隔等）。

这种机制保证了数据的一致性和高效性，能确保在系统崩溃时，未完成的事务不会影响已提交事务的数据完整性。

#### 2. 重做日志（`Redo Log`）

`Redo Log`是`InnoDB`存储引擎中最核心的机制之一，它通过将数据库的物理变更操作记录到日志中，从而实现了高效的写入和恢复机制。`Redo Log`包含两个主要部分：内存中的日志缓存（`Log Buffer`）和磁盘上的日志文件（`Redo Log File`）。它的作用是记录数据库数据页的物理变更，例如：某个特定位置（偏移量`X`）写入了值`Y`。

在事务提交的过程中，`InnoDB`首先会将内存中的日志缓存（`Log Buffer`）中的内容刷新到磁盘上的`Redo Log File`。然后，`InnoDB`通过`fsync`操作将日志文件强制写入磁盘。只有当这些日志成功持久化到磁盘后，事务才被视为已提交。此时，虽然数据库的数据页可能还没有写入磁盘，但是由于`Redo Log`已经被持久化，即使系统发生崩溃，也能通过日志来恢复到上次提交时的数据状态。

因此，`Redo Log`保证了数据库在系统崩溃后仍然能恢复到一致性状态，并且确保了事务的持久性，即使数据页本身还没有被写入磁盘，也可以根据日志恢复操作，确保事务提交后数据的安全性。

#### 3. 二进制日志（`Binlog`）

`Binlog`是记录在`MySQL Server`层的日志，与`InnoDB`的`Redo Log`不同，它属于逻辑日志。根据`binlog_format`的不同，`Binlog`记录的内容可能是执行的增、删、改操作的`SQL`语句（在`STATEMENT`模式下），也可能是数据变更的行记录（在`ROW`模式下）。

`Binlog`的主要用途包括：数据复制（例如主从复制）以及数据恢复（例如使用`mysqlbinlog`工具恢复误删的数据）。需要注意的是，只有在事务提交之后，相关操作才会被写入`Binlog`。

当开启`Binlog`时，事务提交流程会遵循如下顺序：首先写入`Redo Log`，接着写入`Binlog`，两者都写入成功后，事务才被正式提交。这一过程被称为两阶段提交（`2PC`），用于保证`Redo Log`和`Binlog`的一致性。

#### 4. 崩溃恢复机制

当`MySQL`或操作系统崩溃后重启，`InnoDB`会做两件事：

- 用`Redo Log`重做已提交但尚未写入磁盘的数据；
- 用`Undo Log`回滚未提交的事务。

这两个机制保证了数据库恢复时不会丢失已提交的数据，也不会留下未提交的数据。

#### 5. 刷盘策略

`innodb_flush_log_at_trx_commit`参数控制`Redo Log`的刷盘行为，影响数据安全性：

| 参数值 | 行为说明                                  | 安全性 | 性能 |
| ------ | ----------------------------------------- | ------ | ---- |
| `1`    | 每次提交都`fsync`到磁盘                   | 最安全 | 慢   |
| `2`    | 每次提交写入`Log Buffer`，每秒`fsync`一次 | 较安全 | 快   |
| `0`    | 每次提交只写`Log Buffer`，不刷盘          | 不安全 | 快   |

默认参数值是`1`，即每次事务提交都同步写磁盘，这是保证事务不丢失的关键配置之一。
