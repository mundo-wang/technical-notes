更新操作必须结合`Where`条件，以下为常见用法示例。

更新单条记录：

```go
db.Model(&dao.UserInfo{}).Where("id = ?", 3).Update("user_name", "smithsss")
```

基于指定条件进行批量更新：

```go
db.Model(&dao.UserInfo{}).Where("age > ?", 30).Update("age", gorm.Expr("age - 5"))
```

在这里通过`gorm.Expr`构造表达式，实现原值计算。

使用`map[string]interface{}`更新多个字段：

```go
db.Model(&dao.UserInfo{}).
    Where("age >= ?", 30).
    Updates(map[string]interface{}{"email": "temp@126.com", "age": 29})
```

使用结构体更新，这里会忽略零值字段，只对`Email`和`Age`两个字段做更新：

```go
db.Model(&dao.UserInfo{}).
    Where("age >= ?", 28).
    Updates(&dao.UserInfo{Email: "temp18@126.com", Age: 18})
```

上面的`db.Model`也可以换成`db.Table`，直接指定表名字符串`"user_info"`。

`Update`与`UpdateColumn`、`Updates`与`UpdateColumns`在功能上类似，但底层行为存在差异：

- `Update`/`Updates`：会更新整条记录，包含自动维护字段（如`updated_at`），并触发`Gorm`钩子方法。
- `UpdateColumn`/`UpdateColumns`：仅更新指定列，不更新`updated_at`，通常不会触发钩子方法，属于更轻量级操作。

例如执行`Updates`操作语句：

```go
db.Model(&dao.UserInfo{}).Where("age = ?", 20).Updates(&dao.UserInfo{Email: "temp2000@126.com", Age: 18})
```

它实际执行的`SQL`是：

```sql
UPDATE "user_info" SET "updated_at"='2023-12-24 15:35:05.104',"email"='temp2000@126.com',"age"=18 WHERE age = 20 AND "user_info"."deleted_at" IS NULL
```

而执行`UpdateColumns`操作语句：

```go
db.Model(&dao.UserInfo{}).Where("age = ?", 20).UpdateColumns(&dao.UserInfo{Email: "temp2000@126.com", Age: 18})
```

它实际执行的`SQL`是：

```sql
UPDATE "user_info" SET "email"='temp2000@126.com',"age"=18 WHERE age = 20 AND "user_info"."deleted_at" IS NULL
```

根据主键`id`删除单条记录：

```go
db.Delete(&dao.UserInfo{}, 2)
```

根据主键`id`删除多条记录：

```go
db.Delete(&dao.UserInfo{}, []int{1, 2, 3})
```

根据`Where`条件删除数据：

```go
db.Where("age > ?", 18).Delete(&dao.UserInfo{})
```

软删除机制说明：

若模型中包含字段`DeletedAt`，且类型为`gorm.DeletedAt`，则删除操作默认转为软删除，即执行`UPDATE`而非`DELETE`：

```sql
UPDATE "user_info" SET "deleted_at"='2023-12-24 16:17:38' WHERE age > 18 AND "user_info"."deleted_at" IS NULL
```

需要注意的是，软删除会保留数据，仅通过`deleted_at`标记删除状态。后续查询默认自动过滤`deleted_at IS NOT NULL`的数据。

该设计存在潜在风险：若表中存在唯一索引字段（如用户名、手机号），即使执行软删除，这些字段的唯一约束仍然生效。此时，被软删除的数据依然占用唯一值，新数据在插入相同值时会触发唯一约束冲突，从而引发业务异常。因此，在实际设计中，应避免在同一张表中同时使用「唯一索引 + 软删除」的组合方案。

如果想物理删除对应数据，可以使用`Unscoped`方法进行删除：

```sql
db.Unscoped().Where("age > ?", 18).Delete(&dao.UserInfo{})
```

它内部执行的`SQL`为：

```sql
DELETE FROM "user_info" WHERE age > 18
```
