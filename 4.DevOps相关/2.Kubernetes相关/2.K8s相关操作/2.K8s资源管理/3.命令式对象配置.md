命令式对象配置通过`YAML`文件描述资源的具体配置，再以`create`、`delete`、`replace`等命令显式驱动操作。相比纯命令行，配置意图以文件形式固化下来，可纳入版本控制，也便于在团队间传递与审查。

### 一、`YAML`文件结构

`Kubernetes`中每一份资源描述文件，无论类型如何，结构都由以下`4`个顶层字段构成：

- `apiVersion`：资源所属的`API`组与版本，不同资源类型的值不同，例如`Pod`使用`v1`，`Deployment`使用`apps/v1`。
- `kind`：资源类型，例如`Pod`、`Deployment`、`Service`等。
- `metadata`：资源元数据，包含`name`、`namespace`、`labels`等标识信息。
- `spec`：资源的期望状态描述，是配置的核心内容，不同资源类型的`spec`结构各不相同。

以一个最简单的`Pod`配置为例：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  namespace: dev
spec:
  containers:
    - name: nginx
      image: nginx:1.17.1
      ports:
        - containerPort: 80
```

### 二、查阅`YAML`字段说明

在实际编写`YAML`文件时，不同资源的`spec`字段往往较为复杂，可通过以下命令查询任意字段的说明，避免翻文档：

```bash
# 查看Pod下所有顶层字段
kubectl explain pod

# 查看Pod中spec字段的子项说明
kubectl explain pod.spec

# 查看容器配置项
kubectl explain pod.spec.containers
```

`explain`支持任意深度的路径查询，遇到陌生字段时可以一路往下钻：

```bash
kubectl explain pod.spec.containers.resources
kubectl explain pod.spec.containers.livenessProbe
```

### 三、常用操作示例

以下基于同一份`YAML`文件，演示命令式对象配置的完整操作流程，包括命名空间准备、资源创建、查看、更新与删除。

首先创建`dev`命名空间：

```bash
kubectl create ns dev
```

准备`pod.yaml`，描述一个运行`nginx:1.17.1`的`Pod`：

```yaml
# pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  namespace: dev
  labels:
    app: nginx
spec:
  containers:
    - name: nginx
      image: nginx:1.17.1
      ports:
        - containerPort: 80
```

使用`create`命令基于文件创建资源，若资源已存在则报错：

```bash
kubectl create -f pod.yaml
```

查看`Pod`启动状态，加`-o wide`可同时看到所在节点与`Pod IP`：

```bash
kubectl get pods -n dev
kubectl get pods -n dev -o wide
```

查看`Pod`详细信息，包括事件记录、容器状态、挂载路径、调度节点等，排查问题时首选此命令：

```bash
kubectl describe pod nginx-pod -n dev
```

以`yaml`格式查看资源的完整定义，包含`Kubernetes`自动补全的所有默认字段，便于了解实际生效的配置：

```bash
kubectl get pod nginx-pod -n dev -o yaml
```

若需要更新资源，例如将镜像升级至`nginx:1.25`，直接修改`pod.yaml`中的`image`字段，然后使用`replace`命令全量替换：

```bash
kubectl replace -f pod.yaml
```

需要注意的是，`Pod`本身是不可变对象，`replace`实际上是先删除再重建，因此某些结构性字段，例如`spec.containers[*].name`、`volumes`等如果发生变化同样会导致操作失败，日志会明确提示哪个字段不允许修改。

在生产环境中，更新`Pod`通常通过`Deployment`滚动发布来完成，而非直接操作`Pod`资源。

若配置改动较大，可以用`--force`强制替换，其效果等同于先`delete`再`create`，会导致`Pod`短暂中断：

```bash
kubectl replace --force -f pod.yaml
```

删除资源时通过文件指定，语义清晰，不容易误删：

```bash
kubectl delete -f pod.yaml
```

默认会等待资源完全终止后再返回，加`--grace-period=0 --force`可跳过优雅退出直接强删：

```bash
kubectl delete --grace-period=0 --force -f pod.yaml
```

### 四、多资源文件管理

实际项目中，一个应用往往由多种资源组成，例如`Deployment`、`Service`、`ConfigMap`同时存在。可以将它们写在同一个`YAML`文件里，并使用`---`分隔：

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: dev
---
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  namespace: dev
  labels:
    app: nginx
spec:
  containers:
    - name: nginx
      image: nginx:1.17.1
      ports:
        - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: nginx-svc
  namespace: dev
spec:
  selector:
    app: nginx
  ports:
    - port: 80
      targetPort: 80
      nodePort: 30080
  type: NodePort
```

使用`create`、`delete`时同样只需指向这一个文件，其中所有资源会按顺序依次处理：

```bash
kubectl create -f nginx-app.yaml
kubectl delete -f nginx-app.yaml
```

也可以把不同资源各自存为独立文件，统一放在一个目录下，通过`-f`指定目录路径批量操作：

```bash
# 目录结构示例
manifests/
├── namespace.yaml
├── pod.yaml
└── service.yaml

kubectl create -f manifests/
kubectl delete -f manifests/
```

目录方式下，`Kubernetes`会对目录内所有文件按文件名字母序依次处理，若资源之间存在依赖关系（如`Pod`依赖`Namespace`先存在），可在文件名前加数字前缀控制顺序，例如`01-namespace.yaml`、`02-pod.yaml`。

### 五、`YAML`与命令行的换算

在实际使用中，有时需要基于一个已知的命令行操作快速生成对应的`YAML`文件，可以利用`--dry-run=client -o yaml`组合实现，不会真正创建资源，只输出`YAML`内容：

```bash
# 生成Pod的YAML
kubectl run nginx-pod --image=nginx:1.17.1 --port=80 --dry-run=client -o yaml

# 生成Deployment的YAML
kubectl create deploy nginx-demo --image=nginx:1.17.1 --replicas=2 --dry-run=client -o yaml

# 生成Service的YAML
kubectl expose deploy nginx-demo --port=80 --target-port=80 --type=NodePort --dry-run=client -o yaml
```

将输出重定向到文件，即可得到一份可直接使用的基础模板，再按需补充字段：

```bash
kubectl create deploy nginx-demo --image=nginx:1.17.1 --replicas=2 --dry-run=client -o yaml > deploy.yaml
```

这是日常编写`YAML`最高效的方式之一，比从零手写省去大量查字段的时间。
