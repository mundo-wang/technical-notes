安装完`K8s`集群后，若过段时间执行`kubectl get nodes`发现某台`node`节点状态变为`NotReady`：

```sh
NAME     STATUS     ROLES           AGE     VERSION
master   Ready      control-plane   6d21h   v1.28.15
node1    Ready      worker          6d21h   v1.28.15
node2    NotReady   worker          6d21h   v1.28.15
```

我们在`master`节点执行`kubectl describe node node2`，在`Conditions`部分可以看到如下信息：

```sh
MemoryPressure   Unknown   NodeStatusUnknown   Kubelet stopped posting node status.
DiskPressure     Unknown   NodeStatusUnknown   Kubelet stopped posting node status.
PIDPressure      Unknown   NodeStatusUnknown   Kubelet stopped posting node status.
Ready            Unknown   NodeStatusUnknown   Kubelet stopped posting node status.
```

说明`node2`上的`kubelet`已停止向`API Server`上报心跳。

登录`node2`终端，执行`systemctl status kubelet`，发现`kubelet`进程本身是运行中的。执行下面命令查看日志：

```sh
journalctl -u kubelet --no-pager -n 50 | grep -E "Unable to register|Error|error|failed|Failed"
```

发现日志中持续出现如下报错：

```sh
node "node2" is not allowed to modify node "localhost.localdomain"
```

这表示`kubelet`实际使用`localhost.localdomain`作为节点名称，而非`node2`，由于节点只允许操作与自身证书名称完全一致的资源，导致`K8s`节点鉴权拒绝其操作。

执行`hostnamectl`命令查看主机名状态，发现问题根源：

```sh
Static hostname: (unset)
Transient hostname: node2
```

命令执行结果中显示，`Static hostname`未设置，系统开机初期短暂使用默认值`localhost.localdomain`。由于`kubelet`配置了开机自启，它在`systemd`完成主机名初始化之前就已启动，读取到了错误的主机名并完成初始化，后续一直沿用`localhost.localdomain`向`API Server`上报，从而引发权限错误，节点持续`NotReady`。

在`node1`和`node2`上分别执行`hostnamectl set-hostname`，将主机名持久化写入`/etc/hostname`：

```sh
# 在node1上执行
hostnamectl set-hostname node1
# 在node2上执行
hostnamectl set-hostname node2
```

然后在`node2`上重启`kubelet`，使其以正确的主机名重新初始化：

```sh
systemctl restart kubelet
```

等待约一分钟后，在`master`上执行`kubectl get nodes`，即可看到`node2`恢复`Ready`状态。
