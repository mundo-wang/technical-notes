`Shell`脚本是由一系列`Shell`命令组成的文本文件，主要用于自动化执行系统操作。脚本文件通常以`.sh`为后缀，由`Shell`解释器按行解析并执行。`Shell`脚本常见的执行方式有两种：一种是通过`sh`命令执行，另一种是通过`./`直接执行。两者的核心区别在于是否遵循脚本开头的`shebang`（解释器声明，例如`#!/bin/bash`或`#!/bin/zsh`），区别如下所示：

- 使用`sh script.sh`执行脚本时，实际调用的是`/bin/sh`解释器，脚本内容会作为参数传递给它执行。由于解释器已经在命令行中被显式指定，系统不会再去解析脚本文件头部的`shebang`。同理，也可以使用`bash script.sh`命令执行脚本。
- 使用`./script.sh`执行脚本时，系统会根据脚本第一行的`shebang`来调用对应的解释器，脚本会在预期的环境下运行。

例如，如果脚本依赖于`bash`特有的一些语法或特性，却使用`sh script.sh`来执行，就可能因为`sh`与`bash`在兼容性上的差异而出错；而通过`./script.sh`方式执行时，系统会依据脚本开头的`#!/bin/bash`来调用`bash`解释器，使脚本能够按预期正常运行。

另外，通过`sh`命令执行脚本时，不要求脚本本身具备执行权限；而使用`./`方式直接执行脚本时，脚本必须具有可执行权限，否则会报`Permission denied`错误，因此需要先通过`chmod +x script.sh`命令为脚本赋权：

```sh
chmod -x restart.sh   # 移除执行权限
sh restart.sh         # 可以正常执行
./restart.sh          # 会报错：Permission denied
```

需要注意的是，`shebang`必须位于脚本的第一行，并且行首不能包含任何空格或注释，否则系统将无法正确识别并解析解释器。也就是说，`#!`必须作为脚本文件的前两个字符出现。

若需要在`Go`中执行外部命令（包括`sh`脚本或单条系统命令），通常可以通过标准库`os/exec`来实现。`os/exec`用于在`Go`程序中调用操作系统命令，例如执行`ls`、`grep`、`curl`等系统命令，运行`Shell`脚本，启动第三方程序，或作为系统工具的封装执行器使用。其本质是对底层系统调用的一层高级封装，旨在以统一的方式在不同操作系统上创建、执行并管理外部进程。

`os/exec`的核心是结构体`Cmd`，它表示一个准备执行的外部命令，其结构如下所示：

```go
type Cmd struct {
	Path   string
	Args   []string
	Env    []string
	Dir    string
	Stdin  io.Reader
	Stdout io.Writer
	Stderr io.Writer
	...
}
```

创建`Cmd`的对象，需要使用`exec.Command`函数，其函数签名如下所示：

```go
func Command(name string, arg ...string) *Cmd
```

该函数会调用`LookPath`自动查找命令路径，初始化`Cmd`结构，并填充参数数组。其使用示例如下：

```go
cmd := exec.Command("ls", "-l")
```

函数`LookPath`的功能与在`Linux`终端执行`which`命令类似，其函数签名如下所示：

```go
func LookPath(file string) (string, error)
```

其使用示例如下所示：

```go
path, err := exec.LookPath("ls")
```

例如，当传入`ls`时，`LookPath`会按照`PATH`环境变量中配置的目录顺序进行查找。一旦找到对应的可执行文件，就会返回完整路径，例如`/bin/ls`。如果在所有目录中都未找到该命令，则会返回`exec: "xxx": executable file not found in PATH`的错误。

对于使用`Command`函数创建的`cmd`对象，`os/exec`库提供三类执行模式，它们都会阻塞等待命令执行完成。如果命令启动失败、以非`0`状态退出，或被信号终止，都会返回`error`。

首先是调用`Run()`方法，如下所示：

```go
err := cmd.Run()
```

这种方式默认不会返回输出数据，适用于只关心命令是否执行成功的场景。

我们也可以提前绑定输出流，来接收标准输出或标准错误的内容：

```go
var buf bytes.Buffer
cmd.Stdout = &buf
err := cmd.Run()
```

如果需要非阻塞执行，可以使用`Start()`和`Wait()`函数：

```go
err := cmd.Start() // 启动命令执行，但不阻塞
// 可以执行其他逻辑
err = cmd.Wait() // 阻塞等待进程退出
```

命令启动失败时，会在`Start()`函数返回`error`，而非`0`状态退出或被信号终止时，会在`Wait()`函数返回`error`。

调用`Output()`方法，可以获取命令的标准输出，如下所示：

```go
out, err := cmd.Output()
```

这种方式会返回命令执行的`Stdout`，`Stderr`不会返回，当命令以非`0`退出时，返回的`err`会包含`Stderr`的输出内容。

调用`CombinedOutput()`方法，会同时获取命令的标准输出和标准错误，如下所示：

```go
out, err := cmd.CombinedOutput()
```

`CombinedOutput`不会先分别收集`Stdout`和`Stderr`再进行拼接，而是将它们绑定到同一个输出目标，按命令执行时实际写入操作系统的顺序实时合并。因此，看到的输出顺序与命令执行的实际执行顺序一致。

如果想让命令的输出会直接打印到当前终端，可以使用下面的方式：

```go
cmd.Stdout = os.Stdout
cmd.Stderr = os.Stderr
```

如果想实现管道操作，例如`ls -l | grep "txt"`，需要手动使用`os/exec`创建两个`Cmd`对象，并通过`io.Pipe`将前一个命令的`Stdout`连接到下一个命令的`Stdin`。代码如下所示：

```go
cmd1 := exec.Command("ls", "-l")
cmd2 := exec.Command("grep", "txt")
// 创建管道，将cmd1的输出连接到cmd2的输入
pipeReader, pipeWriter := io.Pipe()
cmd1.Stdout = pipeWriter
cmd2.Stdin = pipeReader
var output bytes.Buffer
cmd2.Stdout = &output
_ = cmd1.Start()
_ = cmd2.Start()
_ = cmd1.Wait()
_ = pipeWriter.Close()
_ = cmd2.Wait()
fmt.Println(output.String())
```

如果想执行项目下的脚本文件，也可以使用`os/exec`库来完成。例如项目结构如下：

```scss
project
├─ scripts
│  └─ restart.sh
└─ main.go
```

在`Go`中创建`Cmd`时，直接指定脚本的路径即可，例如：

```go
cmd := exec.Command("./scripts/restart.sh")
cmd.Stdout = os.Stdout
cmd.Stderr = os.Stderr
_ := cmd.Run()
```

这需要保证在部署项目到`Linux`服务器时，必须将`restart.sh`一并拷贝过去，并保持其位于可执行文件所在目录下的`scripts`子目录中，否则程序在运行时将无法找到该脚本文件并执行。

如果希望通过`sh`命令来执行脚本文件，则需要在创建`cmd`对象时改为显式指定`sh`解释器：

```sh
cmd := exec.Command("sh", "./scripts/restart.sh")
```
