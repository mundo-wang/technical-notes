索引是用于提高数据库表的检索效率的数据结构，通过创建索引，MySQL可以快速定位到特定行，而不用扫描整张表，同时也降低了数据排序的成本，降低CPU消耗。

创建索引：

```sql
create index 索引名 on 表名(字段名);
```

查看表中所有索引：

```sql
show index from 表名; 
```

删除索引：

```sql
drop index 索引名 on 表名;
```

查看表中某一字段是否有索引

```sql
SHOW INDEX FROM 表名 WHERE Column_name = '字段名';
```

如果有索引，会展示具体索引信息；如果无索引，展示空值。

索引缺点：

1. 索引需要额外的存储空间。
2. 索引降低了表插入、更新、删除的效率，不适合用于频繁进行数据更新的表。
3. 在写操作频繁的情况下可能会引发锁竞争问题。

MySQL底层使用B+树来作为索引的数据结构。B+树的优点是：

1. B+树是一种自平衡的树结构，保证在插入和删除操作时，树的高度保持在一个相对稳定的范围。
2. B+树可以保持索引数据的有序性，且叶子结点间连接成链表，使得范围查询和排序的效率很高。
3. 由于B+树的平衡性和有序性，可以减少与磁盘的IO操作，提高查询效率。
4. B+树的结构可以很容易扩展到支持更大的数据集，适用于大型存储系统。

MySQL的索引是越多越好吗？

并不是。索引过多的缺点如下：

1. 每次对表进行写操作（插入、更新、删除）都需要更新索引，索引太多，会导致数据插入的性能降低。
2. 过多的索引会导致数据库占用更多的磁盘空间。
3. 索引需要加载到内存中以提高查询性能，而过多的索引可能导致内存占用过高。
4. 过多的索引也会导致查询性能下降，因为数据库引擎需要花时间评估多个可能的索引。
5. 随着索引增加，数据库的维护成本也会增加，包括备份、恢复、数据迁移等操作。
