### 1. ID的要求 
在分布式微服务或云原生环境中，对于数据库表的主键有一些重要的要求。

主键id的需要保证的一些要求：
1. 全局唯一
    主键必须在整个系统是唯一的，避免数据冲突和不一致，这是最基本的要求。
2. 趋势递增
    MySQL使用InnoDB引擎，使用的是聚簇索引，所以主键尽量使用有序的。
3. 单调递增
    保证下一条数据的id一定大于上一个id。
4. 信息安全
    如果id是连续的，容易被恶意爬取，如果是订单号的表，别人可以轻而易举知道订单数量。所以在一些场景下需要id无规则。
5. 含时间戳
    能在开发中快速了解分布式id生成的时间。

### 2. UUID
标准型式包含32个16进制数字，以连字号分为五段，形式为8-4-4-4-12的32个字符。

UUID优点：

1. UUID的性能非常高，它在本地就能生成，没有网络消耗。
2. 确保唯一性的能力较强，避免数据冲突问题。
3. UUID基于时间戳和随机数生成，不易被猜测或破解。

UUID缺点：

1. UUID入数据库的能力较差，因为UUID是无序的，插入时会破坏原本索引B+树的结构。
2. UUID太长，作为主键的话，索引表占用存储空间比较大，如果数据量很大，就要考虑存储量的问题。
3. UUID往往以字符串形式存储在数据库，查询的效率比较低。


### 3. 自增主键
在数据库中创建一个自增长的值，每次插入新纪录时自动递增。

自增主键优点：

1. 简单易用，不需要开发者管理和维护主键的生成。
2. 减少插入操作导致的额外索引维护开销。

自增主键缺点：

1. 可能会导致ID冲突问题。
2. 系统水平扩展困难：如果做分库分表，假如第一张表已经排到了1000，下一张表的起始值要设置多少呢？10000？两张表还好说，如果是10张表呢？所以系统水平扩展几乎很难做到。第二点就是，如果要与其他系统集成，涉及到合表操作等时，很容易造成主键冲突。
3. 单点故障，如果自增序列所在节点出现故障，整个系统将无法继续生成主键，导致写入操作受影响。

### 4. 雪花算法
用于生成分布式系统唯一ID的算法，并且此ID按照时间有序递增。由64bit大小的整数，由1bit的符号位、41bit的时间戳、10bit的工作机器id、12bit序列号组成。

雪花算法优点：

1. 生成ID性能高
2. ID能保证全局唯一
3. ID按照时间顺序递增，方便排序和索引。

雪花算法缺点：

1. 依赖机器时钟，如果机器时钟回拨，会导致有重复id生成。
2. 机器ID的位数限制了系统中可部署机器的数量。
3. 序列号的位数限制了同一毫秒内生成的ID数量，如果并发量超过限制，可能有id冲突。


### 5. 如何选择
1. UUID：适合分布式系统和需要保证唯一性的场景。
2. 自增主键：适合单节点或者较小规模的系统。
3. 雪花算法：适合分布式系统和需要保证唯一性和顺序性的场景。
