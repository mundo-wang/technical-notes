`IPv4`的`32`位地址在理论上可提供`2^32`（约`43`亿）个地址。然而，受协议设计、保留地址段以及实际网络规划等因素影响，真正可用于分配的公网地址数量远低于这一理论上限。面对全球数十亿人口以及规模达数百亿的联网设备，早已无法满足需求。

`IPv6`的全称为`Internet Protocol version 6`，是为缓解`IPv4`地址枯竭问题而设计的下一代互联网协议。`IPv6`采用`128`位地址空间，理论上可提供`2^128`个地址，总量约为`3.4 × 10^38`，在可预见的未来范围内可以视为几乎不可能耗尽。

`IPv6`地址采用`16`进制表示，将`128`位划分为`8`组，每组`16`位（即`4`个十六进制字符），各组之间以`:`分隔。例如：

```
2001:0db8:85a3:0000:0000:8a2e:0370:7334
```

为便于书写与阅读，`IPv6`支持如下压缩规则：

1. 每组中的前导`0`可以省略，例如`0db8`可简写为`db8`。
2. 连续的全`0`分组可以使用`::`替代，但在同一个地址中仅允许出现一次，以避免歧义。

基于上述规则，示例地址可压缩为：

```
2001:db8:85a3::8a2e:370:7334
```

在`IPv6`中，不再使用像`255.255.255.0`这种掩码表示法，而是统一使用前缀长度，例如：

```
2001:db8:abcd:0012::/64
```

在实际部署中，`IPv6`几乎统一使用`/64`作为子网前缀长度，即地址的前`64`位表示网络部分，后`64`位表示接口标识。接口标识既可以根据`EUI-64`规则由网卡的`MAC`地址生成，也可以通过随机或稳定随机算法自动生成。

在一个`/64`子网中，理论上可以容纳`2^64`个接口地址。与`IPv4`不同，`IPv6`不存在传统意义上的“网络地址”和“广播地址”概念，因此不需要像`IPv4`那样预留并扣除`2`个特殊地址。

由于`IPv4`与`IPv6`在报文结构与协议机制上存在根本性差异，二者无法直接互通，因此在实际部署中通常采用以下过渡方案：

1. 双栈（`Dual Stack`）：设备同时运行`IPv4`与`IPv6`协议栈，根据目标地址类型选择对应协议进行通信。
2. 隧道技术（如`6to4`）：在`IPv4`网络中对`IPv6`数据包进行封装传输，实现跨`IPv4`基础设施的`IPv6`通信。
3. 协议转换（如`NAT64`）：通过地址与报文转换机制，实现`IPv6`网络与`IPv4`网络之间的互通。

在当前互联网环境中，主流操作系统与云服务提供商已默认支持`IPv6`。整体趋势是以`IPv6`为核心构建网络架构，而`IPv4`则通过转换与兼容机制在较长时期内继续共存。
