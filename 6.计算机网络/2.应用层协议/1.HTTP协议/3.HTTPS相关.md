`HTTP`被认为是不安全的协议，主要原因有三个：

1. 明文传输：`HTTP`以明文传输数据，通信内容未加密，任何监听网络的第三方都可能读取数据，导致敏感信息泄露。
2. 身份无法验证：`HTTP`无法验证通信双方身份，客户端无法确认连接的服务器是否真实可信，容易遭受中间人攻击（`MITM`）。
3. 数据易被篡改：`HTTP`缺乏数据完整性校验机制，攻击者可在传输过程中修改请求或响应内容，而通信双方无法发现数据已被篡改。

为了解决这些问题，引入了`HTTPS`技术。`HTTPS`并不是一种全新的应用层协议，而是在`HTTP`与`TCP`之间增加了一层`TLS`协议，使`HTTP`报文能够在加密通道中安全传输，如下图所示：

尽管`SSL`（`Secure Sockets Layer`，安全套接层）协议早已被`TLS`（`Transport Layer Security`，传输层安全）协议取代，但在实际工程中，仍会习惯使用`SSL`或`SSL/TLS`作为这一类安全协议的统称。因此，日常所说的`SSL`证书，本质上是符合`X.509`标准的数字证书。事实上，现代`HTTPS`通信普遍采用的协议版本均为`TLS 1.2`或`TLS 1.3`。

`HTTPS`主要提供以下三个方面的安全保障：

1. 加密传输：`HTTPS`在数据传输前，会通过`TLS`协议对通信内容进行加密。即使数据在传输过程中被第三方截获，由于其内容为密文，攻击者也无法直接解析出真实信息，从而有效防止敏感数据泄露。
2. 身份认证：`HTTPS`依赖数字证书体系。服务器会向客户端提供由权威`CA`机构签发的数字证书，客户端通过校验证书的合法性，确认当前访问的服务器身份真实可信，从而避免连接到伪造服务器，降低中间人攻击的风险。
3. 数据完整性校验：`TLS`会对传输的数据进行完整性保护。接收方能够验证数据是否在传输过程中被篡改，一旦校验失败，将立即丢弃该数据，确保通信内容的完整性。

`HTTPS`协议下，浏览器与服务器需要先完成`TLS`握手，协商出共享的会话密钥，随后再建立加密通信。典型流程如下：

1. 客户端发送`ClientHello`：浏览器向服务器发送握手请求，携带自身支持的`TLS`版本、加密套件列表、客户端随机数等信息。
2. 服务器响应`ServerHello`：服务器选择双方都支持的一种加密套件，返回服务器随机数，同时发送服务器数字证书。证书中包含服务器公钥、域名、有效期以及签发机构等信息。
3. 客户端验证服务器证书：浏览器会对服务器返回的数字证书进行校验，包括证书链是否可信、证书是否过期、域名是否匹配以及证书是否被吊销等，确认服务器身份真实可信。
4. 密钥协商：证书验证通过后，客户端与服务器根据协商好的密钥交换算法（现代`TLS`通常采用`ECDHE`），结合客户端随机数、服务器随机数以及密钥交换过程中产生的共享秘密，共同生成本次连接独有的会话密钥。
5. 建立加密通信：握手完成后，双方切换至加密状态，后续所有`HTTP`请求与响应均使用协商出的会话密钥进行对称加密传输。

需要注意的是，数字证书的主要作用并不是加密业务数据，而是证明服务器身份，并安全地向客户端传递服务器公钥。真正用于加密业务数据的是握手过程中协商生成的会话密钥。

现代`HTTPS`通信并不会一直使用非对称加密。由于非对称加密计算开销较大，仅适用于身份认证和密钥协商；真正的数据传输阶段采用的是效率更高的对称加密算法，例如`AES-GCM`或`ChaCha20-Poly1305`。这种设计兼顾了通信安全性与传输性能。

在`TLS`握手过程中，客户端随机数与服务器随机数均为`32`字节，它们会与密钥交换过程中生成的共享秘密共同参与会话密钥的生成。由于每次握手都会生成新的随机数，即使使用相同的数字证书和加密套件，每次连接生成的会话密钥也都不同，从而保证了密钥的唯一性与不可预测性。现代`TLS`默认采用`ECDHE`等支持前向保密（`Forward Secrecy`）的密钥交换算法。这意味着即使服务器的长期私钥在未来泄露，攻击者也无法解密此前抓取的通信数据，因为每一次连接都会生成独立的临时会话密钥。

`HTTP`与`HTTPS`的特征比较如下：

| 特征       | `HTTP`                         | `HTTPS`                                                      |
| ---------- | ------------------------------ | ------------------------------------------------------------ |
| 安全性     | 数据以明文传输，不安全。       | 数据经过加密，提供安全连接。                                 |
| 协议       | 超文本传输协议。               | 基于`HTTP`与`TLS`构建的安全通信协议。                        |
| 默认端口   | `80`端口。                     | `443`端口。                                                  |
| 身份认证   | 不支持。                       | 通过数字证书验证服务器身份。                                 |
| 数据完整性 | 不提供完整性保护。             | 提供完整性校验，能够检测数据是否被篡改。                     |
| 连接方式   | 非安全连接。                   | 使用`TLS`建立安全连接。                                      |
| 数据加密   | 不加密。                       | 使用对称加密保护数据传输。                                   |
| 性能       | 无需握手和加密，性能开销较低。 | 首次连接需完成`TLS`握手和加解密，存在一定性能开销，但现代硬件和协议优化后影响较小。 |
| 成本       | 无需部署数字证书。             | 需要部署和维护数字证书，证书既可以免费获取，也可以购买商业证书。 |
| 配置       | 配置相对简单。                 | 配置相对复杂，涉及数字证书和`TLS`参数管理。                  |

综上，`HTTPS`通过`TLS`协议提供了加密传输、身份认证以及数据完整性保护，有效弥补了`HTTP`在安全性方面的天然缺陷。目前，`HTTPS`已经成为互联网应用事实上的标准通信方式。
