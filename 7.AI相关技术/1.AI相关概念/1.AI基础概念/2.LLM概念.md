`LLM`的全称是`Large Language Model`，中文通常翻译为「大语言模型」，简称大模型。

目前几乎所有主流大模型都基于`Transformer`模型。`Transformer`架构的核心在于`Self-Attention`（自注意力机制）。

在传统的循环神经网络（`RNN`）中，模型处理句子时，必须先处理前一个词，才能继续处理下一个词。这种机制容易导致模型「遗忘」句首信息。而`Self-Attention`具有全局视野，在处理每个`Token`时都会同时关注句子中的所有其他词。

以句子「那个奖杯放不进手提箱，因为它太大了」为例，模型在处理「它」时，往往会为「奖杯」分配较高的注意力权重。这是因为在常识语义中，「大」通常是导致「放不进」的直接原因，而「放不进」的主体通常是被放置的物体，所以模型倾向于将「它」解析为「奖杯」。若将句子改为「那个奖杯放不进手提箱，因为它太小了」，模型在处理「它」时，注意力权重则更可能转移到「手提箱」。在常识语义中，容器过小才是造成空间不足的原因，因此模型会倾向于将「它」解析为「手提箱」。

`Self-Attention`的核心计算过程是：每个`Token`首先被映射为三组向量：`Query`、`Key`和`Value`。随后通过计算`Query`与`Key`之间的点积获得注意力权重，并将该权重作用于对应的`Value`向量进行加权求和，从而实现对上下文信息的动态聚合。

> 在计算机中，向量通常表示为由一组有序、同类型元素构成的一维数据序列。从数学角度看，向量通常被表示为列向量，但在编程实现中通常不会区分行向量与列向量，而仅以一维结构（数组或切片）存储。

大模型通常需要经历一个规模巨大的训练阶段，这一过程称为预训练（`Pretraining`）。其核心目标是在海量文本数据上学习语言的统计规律与上下文依赖关系。预训练通常采用自监督学习（`Self-Supervised Learning`）范式，通过构造无需人工标注的训练任务，使模型在不断最小化预测误差的过程中逐步将语言知识编码进参数之中，从而获得基础的语言理解与生成能力。

神经网络本质上是一个极其复杂的函数，输入是经过编码的数字序列，输出是下一个词的概率分布。

我们从最简单的线性函数出发，以`y = ax + b`为例，其中`a`和`b`就是参数，给定这两个数，函数的行为便完全确定，输入任意`x`都能算出对应的`y`。神经网络遵循同样的逻辑，只是参数数量从`2`个扩展到了数百亿个，这些参数的集合就是「权重」（`Weights`）。

权重在训练之前是随机初始化的，此时模型什么也不会。预训练的本质，就是用海量文本不断调整这数百亿个数值，使模型的输出不断逼近正确答案。训练结束后，这些数值固定下来，以多维数组（张量）的形式保存为二进制文件，即权重文件（`Model Weights`）。权重是训练过程积累的所有知识的压缩形态，语法规律、事实知识、推理模式，全部以数值的形式分散编码在参数里。

有了权重文件，加上对应的模型架构代码，便能在本地还原出完整的模型并执行推理，无需重新训练。

预训练的核心前提是确定模型参数总量（即日常所说的「多少`B`」，`B`即`Billion`，十亿），其大小并非随意设定，而是由硬件能力决定。显存总量、`GPU`数量以及整体集群算力，共同构成了模型规模的上限。以消费级显卡为例，通常只能支撑`7B`、`13B`等中小规模模型；而千亿级（`100B`以上）的模型，则需要上万张专业`GPU`（如`A100`、`H100`）组成的大规模计算集群才能完成训练。

每个参数在存储时会以特定的数值精度（`Precision`）表示，精度的选择直接决定了模型权重文件的体积与运行时的显存占用。

目前常见的精度格式有以下几种：`FP32`（`32`位浮点）每个参数占`4`字节，精度最高，通常用于研究场景；`FP16`与`BF16`（`16`位浮点）每个参数占`2`字节，是当前训练与推理最主流的选择；`INT8`与`INT4`（量化精度）每个参数分别占`1`字节与`0.5`字节，文件体积显著缩小，但会引入一定程度的精度损失。

将参数量与每个参数占用字节数相乘，即可估算出权重文件的大小。以`7B`模型为例：以`FP16`存储时，文件体积约为`14GB`；经`INT4`量化后，则可压缩至约`3.5GB`。这也解释了为何同一模型在`Ollama`或`Hugging Face`上通常会提供多个版本，例如`Q4_K_M`、`Q8_0`、`BF16`等，它们的参数规模完全相同，仅存储精度有所差异，文件大小与所需显存也随之变化。

显存与内存在物理介质和设计目标上存在本质差异，二者在速度与价格上的差距也较为显著。

速度方面，显存（以`GDDR7`或`HBM3`/`HBM3e`为主流）的带宽远高于内存（以`DDR4`/`DDR5`为主流）。以具体数值为例，`H200`所搭载的`HBM3e`显存带宽可达`4.8 TB/s`，而主流`DDR5`内存的带宽通常在`50～90 GB/s`之间，二者相差约`50～90`倍。大模型推理的核心瓶颈正是参数的读取速度而非算力本身，因此显存带宽的高低直接决定了推理的吞吐量与响应延迟。

价格方面，显存的单位容量成本远高于内存。以消费级市场为参照，`DDR5`内存的价格大约在`3～5`美元每`GB`，而搭载`32 GB`显存的`RTX 5090`官方定价为`1999`美元，折合显存单价约为`62`美元每`GB`，是同期内存的`12～20`倍。专业级`GPU`的显存单价则更高，`H200`的市场价格约为`30000～40000`美元，其搭载`141 GB`的`HBM3e`显存，折合显存单价约为`213～284`美元每`GB`。

这一价格差距的根本原因在于显存的制造工艺与封装复杂度远高于普通内存，`HBM`系列显存需要通过硅中介层（`Interposer`）将多层`DRAM`芯片与`GPU`裸片进行三维堆叠封装，良率低、工艺难度大，成本自然居高不下。

模型参数的加载位置取决于运行环境：若使用`GPU`进行推理，参数将加载至显存（`VRAM`）；若使用`CPU`进行推理，或显存不足时触发换出，则参数会加载至内存（`RAM`），或由二者共同承载。权重文件大小与推理时的显存/内存占用在数值上高度接近，原因在于推理时需将全部参数载入显存/内存参与计算，二者本质上是同一份数据在磁盘与显存/内存中的不同存储状态。实际运行时的占用会略高于文件大小，差值主要来自`KV Cache`及推理框架的少量运行时开销。

日常估算时，直接以权重文件大小对照可用显存或内存容量作为判断依据，是合理的经验做法。

确定参数规模之后，还需配套相应规模的预训练语料。行业普遍遵循`Scaling Law`（缩放定律）：在给定算力预算下，参数量与预训练所需`Token`数存在最优比例，参数量越大，所需语料规模也越大。一般来说，`1B`参数对应约`20B Token`。

值得注意的是，以`LLaMA`系列为代表的工程实践普遍采用「过训练」策略，即使用远超理论最优量的语料。该策略主要适用于小模型，例如`7B`模型在该策略下通常使用约`1T`（即`1000B`）`Token`进行训练，以换取更小模型在推理阶段更强的实际表现。而对于`671B`量级的大模型，由于其推理成本本身已极高，激进过训练的收益有限，因此更倾向于遵循`Chinchilla`理论最优配比，以`DeepSeek-V3`为例，其训练语料约为`14.8T Token`，与理论最优值高度吻合。

如果数据规模与模型参数规模不匹配，模型可能无法有效学习，从而出现过拟合或欠拟合问题。参数多而数据少容易过拟合，模型仅记住训练数据，泛化能力差；参数少而数据多容易欠拟合，模型容量不足，无法充分学习知识。

训练数据通常来源于互联网文本、书籍以及代码等多种语料，先通过分词器（`Tokenizer`）将原始文本统一转换为`Token`序列，再映射为`TokenID`整数序列，作为大模型训练输入。模型会预先配置嵌入层（`Token Embedding`），为每个`Token`初始化高维浮点向量（维度通常在`4096`～`16384`之间）。向量的单个维度不对应人工定义的具体语义，而是模型通过海量数据自学习得到的抽象特征。

经过训练迭代优化后，语义或用法相近的`Token`在高维空间中的向量距离更近，而语义差异大的`Token`向量距离更远。`Embedding`后的结果与原始`Token`会一同存储在模型的权重文件（`Model Weights`）中，并在运行时加载至显存（`VRAM`）或内存（`RAM`）。这些向量是静态且只读的，除非对模型进行重新训练，否则其内容不会发生变化。

同时，在训练过程中会生成预训练分词词表，记录`Token`与`TokenID`的映射关系。

模型蒸馏（`Model Distillation`）是一种模型压缩技术，核心思想是让小模型（学生模型）向大模型（教师模型）学习。其关键不在于直接模仿教师的最终答案，而在于学习教师输出的软概率分布。以图像分类为例，教师模型对一张猫的图片输出的不是简单的「猫」这一硬标签，而是「猫`85%`、猎豹`8%`、老虎`4%`……」这样的概率分布，其中隐含了类别之间相似性关系等大量隐性知识。

市面上常见的`DeepSeek-R1-Distill-Qwen-7B`（从`DeepSeek-R1`蒸馏出来的、基于`Qwen`架构的`7B`模型）等模型均通过该技术从更大规模的模型蒸馏而来，这也解释了为何此类小模型在本地部署场景中能够展现出远超其参数规模的实际能力。

预训练结束后所得到的模型称为基座模型（`Base Model`），它只具备语言建模能力，即根据已有上下文预测下一个`Token`。直接运行基座模型权重时，模型的行为就是纯粹的文本接龙，输入「请介绍一下北京」，它可能接续出「的天气如何？」或「故宫的历史……」，而不会给出一个完整的回答。这并非缺陷，而是预训练目标本身决定的结果。

经过微调（`Fine-tuning`）对齐之后，模型才具备理解指令、按预期格式作答的能力，产出的模型通常被称为指令模型（`Instruct Model`）或对话模型（`Chat Model`）。这也是为何在`Hugging Face`或`Ollama`上能看到同一参数规模的模型往往提供两个版本，例如`Qwen3-8B`与`Qwen3-8B-Instruct`，前者是基座，后者是经过微调对齐的可用版本。

微调是将大模型从「通用能力」收敛到「特定任务能力」的核心对齐手段。其本质是在预训练模型之上，引入规模较小但高质量的领域数据进行二次训练，使模型在特定任务（如代码生成、医疗问答、法律建议）上的表现得到定向优化。微调的训练数据不是泛化的知识库，而是经过精心筛选的标准答案集。

微调通常分为两个阶段：有监督微调（`SFT`）与基于人类反馈的强化学习（`RLHF`）。`SFT`通过构造「指令-输出」数据对，使模型学习遵循指令、理解语义结构，并建立对输出格式的稳定预期，例如何时使用列表、何时输出`JSON`、何时给出分步推理；`RLHF`则进一步引入奖励模型（`Reward Model`），对模型生成结果进行打分与排序，从而在多样性候选中筛选出更符合人类偏好与安全约束的输出。

为降低训练成本，当前工程实践中普遍采用参数高效微调（`PEFT`），其中以`LoRA`最具代表性。该方法通过在模型权重旁路引入低秩矩阵，仅训练少量新增参数来建模任务相关信息；训练完成后，这些增量参数可与原模型权重合并，从而实现性能提升。相较于全量微调，该方式显著降低了显存占用与计算开销，使微调能够在有限硬件资源下完成；同时，由于对原始参数的扰动较小，也在一定程度上缓解了「灾难性遗忘」问题，能够在保持模型通用能力的前提下实现有效的领域适配。

在上一节中提到，大模型并不会直接处理用户输入的自然语言文本，而是先通过`Tokenizer`将文本切分为多个`Token`，再根据预训练分词词表将其映射为`TokenID`序列。随后，每个`TokenID`会经过`Embedding`层映射为高维向量，并叠加位置编码，从而形成同时包含语义信息与顺序信息的数值矩阵，作为模型计算的输入。

从底层机制来看，大模型并不具备真正意义上的主观思考与逻辑推理能力，本质上是一种基于`Token`的自回归序列预测模型，通过逐词生成完成文本接续。模型在多层`Transformer`结构中利用自注意力机制对输入的高维向量序列进行计算，最终得到一个高维隐层向量，该向量仅表示语义趋势和上下文逻辑，并非某个现成`Token`对应的向量。

随后，该隐层向量会映射到预训练阶段构建的分词词表空间，并与词表中所有`Token`的向量逐一比对打分，更接近隐层向量的`Token`向量会获得更高分。模型不会直接选择分数最高的`Token`，而是结合温度系数、`Top-P`、`Top-K`等采样策略，在概率较高的候选集合中进行筛选采样，得到更合理且多样化的`TokenID`，最终通过分词器查表解码为具体`Token`文本。

我们可以用三维向量作为示例来说明。假设词表中有三个词：苹果、桃子、手机，它们对应的向量为：

```scss
Token      三维向量
苹果       [0.9, 0.1, 0.2]
桃子       [0.8, 0.2, 0.3]
手机       [0.1, 0.9, 0.7]
```

当模型输入序列「我想吃」后，经过`Embedding`层和`Transformer`计算，得到输出的隐层向量为`[0.85, 0.15, 0.25]`。

随后，模型会通过点积算法，计算该隐层向量与词表中每个`Token`向量的相似度：

```scss
隐层向量·苹果 = 0.85×0.9 + 0.15×0.1 + 0.25×0.2 = 0.765 + 0.015 + 0.05 = 0.83
隐层向量·桃子 = 0.85×0.8 + 0.15×0.2 + 0.25×0.3 = 0.68 + 0.03 + 0.075 = 0.785
隐层向量·手机 = 0.85×0.1 + 0.15×0.9 + 0.25×0.7 = 0.085 + 0.135 + 0.175 = 0.395
```

得到这些分值后，通常会接入`Softmax`函数，将原始得分（`Logits`）转换为概率分布。其过程是先以`e`为底对各分值取指数，再计算指数和，最后完成归一化得到概率。计算结果显示：苹果的概率为`38.41%`，桃子的概率为`36.72%`，手机的概率为`24.87%`。因此模型会判断苹果与当前语义最接近并优先输出对应的`Token`，但在一定采样策略下，也可能选择桃子。

我们可以先不考虑隐层向量与`TokenID`之间的计算与映射过程，只聚焦单个`Token`逐步生成的本质。

例如输入「蒙多想去哪就去哪」，`Tokenizer`会将其转换为`TokenID`序列`[75, 61, 18, 22, 12, 22]`。该序列输入模型后，模型基于已有上下文预测下一个`Token`为「是」，对应的`TokenID`为`32`。此时序列更新为`[75, 61, 18, 22, 12, 22, 32]`。

接着，这个新序列会再次输入模型，继续预测下一个`Token`，如此循环。既然是「接龙」，就必须有一个停止条件。在整个词表中，通常会包含一个特殊的「隐形`Token`」，称为`[EOS]`（`End Of Sequence`，终止符）。当模型在某一步计算中，发现概率最高，或者通过采样选中的`TokenID`正好对应这个`[EOS]`时，就表示当前序列已经完整，生成过程应当结束。

在`Decoding`（解码）阶段，为提升用户体验，当前主流`LLM`通常采用`Stream`（流式传输）机制，即模型每生成一个`Token`，就会立即完成解码并返回给用户，因此在实际使用大模型平台时，会观察到逐步流式输出的效果。

这种`Token`接龙的过程在技术上称为自回归生成（`Autoregressive Generation`）。从工程角度看，从接收指令到完成接龙的整个运行过程，被统称为「推理」（`Inference`）。

大模型会通过`KV Cache`技术，将已计算过的`Token`特征缓存起来；在每一轮生成时，仅需对最新的一个`Token`进行计算，从而避免大规模的重复算力消耗。在`API`计费层面，费用是基于输入总量与生成总量进行统计的，并不会因为循环调用次数而重复计费。

那么，为何各大模型服务商普遍以`token`消耗量作为计费标准？根本原因在于，`token`是大模型推理过程的天然计量单位。输入侧需要对所有`token`完成一次`prefill`（全量注意力计算），输出侧则以自回归方式逐`token`生成，每生成一个`token`都对应一次完整的前向传播。因此，`token`数量与实际`GPU`算力消耗之间存在稳定的线性对应关系，是度量推理成本最直接的载体。

此外，输入`token`与输出`token`在计费时通常单独定价，且输出价格约为输入的`3`至`5`倍，原因在于二者的资源占用模式不同。

输入`prefill`阶段可并行计算，`GPU`利用率高；而输出`decode`阶段是串行的逐`token`生成，`GPU`利用率较低，且随着序列增长，`KV Cache`的显存占用持续累积，单位算力成本更高，因此定价也更贵。

> 该模型的`Hugging Face`地址：https://huggingface.co/Qwen/Qwen3-8B/tree/main

模型权重总计约`16.42 GB`，因单文件体积过大，被拆分为`5`个`.safetensors`分片进行存储。`.safetensors`格式相比传统的`.bin`格式，加载更安全、速度更快，且完全不依赖`pickle`序列化机制，从根本上规避了反序列化漏洞风险。

`model.safetensors.index.json`作为整个权重体系的索引入口，推理框架在加载模型时通过它定位各权重所在的分片文件。

`merges.txt`、`tokenizer.json`、`tokenizer_config.json`、`vocab.json`这`4`个文件共同构成一套完整的`BPE`分词体系。其中`tokenizer.json`是`HuggingFace`格式的完整快速分词器描述，包含词表、合并规则及后处理配置，单独使用即可完成全流程分词；其余`3`个文件则是面向旧版`tokenizers`库的拆分兼容形式，在新版环境中通常无需单独使用。

用户输入到大模型并得到输出的整体流程，其`PlantUML`代码如下所示：

```scss
@startuml
actor 用户
participant "Tokenizer" as Tokenizer
participant "TokenID序列" as Tokens
participant "大语言模型" as LLM
participant "概率分布+采样策略" as Sampling

用户 -> Tokenizer : 输入文本
Tokenizer -> Tokens : 编码，转换为TokenID序列

loop 逐Token生成
    Tokens -> LLM : 输入当前序列
    LLM -> Sampling : 计算概率并采样下一个Token
    Sampling --> Tokens : 返回TokenID，追加到序列

    Tokens -> Tokenizer : 单TokenID解码
    Tokenizer --> 用户 : 流式输出对应文本片段

    alt 已生成EOS
        Tokenizer --> 用户 : 代表已输出完整结果
    else 未生成EOS
        Tokens -> LLM : 输入当前序列，重复上述步骤
    end
end
@enduml
```
