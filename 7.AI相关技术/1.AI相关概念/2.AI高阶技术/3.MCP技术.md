在上一篇内容中，我们讲到平台需要在配置中维护「工具名称到调用地址、认证方式等信息的映射」，并自行解析工具调用格式、拼装`HTTP`请求。这意味着每接入一个新工具，平台侧都需要单独开发对接逻辑，工具与平台之间形成了强耦合的点对点关系。随着工具数量的增长，维护成本会急剧攀升，且同一工具也无法在不同平台间复用。

更深层的问题在于，平台会在`System Prompt`中注入工具的「名称、功能说明与入参格式」，但这份描述如何撰写、采用何种结构，完全由平台自行决定，缺乏统一规范。各平台描述方式不一，导致大模型的理解质量参差不齐。加之工具列表以静态方式固化在平台侧，新增或下线工具都需要修改配置并重新部署，灵活性极为有限。

于是在工程落地时，引入`MCP`（`Model Context Protocol`）来解决这些暴露出的瓶颈：

- 标准化接入：`MCP`定义了一套统一的通信协议，平台侧只需实现一套通用的`MCP Client`逻辑，即可对接所有遵循该协议的工具，彻底消除了重复适配的工程负担。工具侧同样只需按协议实现一次`MCP Server`，便能被任何支持`MCP`的平台直接发现与调用，从根本上打破了工具与平台之间的强耦合关系。
- 规范化描述：`MCP`统一了工具元数据的描述格式，包括名称、功能描述、入参`Schema`及类型约束等，使大模型能够以一致、可预期的方式解析工具能力，显著提升了工具调用的准确性与稳定性。
- 动态工具发现：`MCP`支持`Client`在运行时通过`tools/list`请求向`Server`主动拉取当前可用的工具列表，无需人工干预即可感知工具的新增与下线，真正实现了工具的动态发现与按需更新，大幅降低了工具全生命周期的管理成本。

需要澄清一个常见的误区：许多人认为「大模型将工具调用意图返回给平台」这部分的`JSON`结构也属于`MCP`的规范范畴，但实际上并非如此。这份`JSON`结构属于大模型`API`层面的协议，由各大模型厂商自行定义，不同厂商的响应格式存在差异，这是模型接口层面的问题，不在`MCP`的管辖范围之内。`MCP`规范的仅是「平台如何与工具交互」这一环节。

`MCP`规范将参与方分为`Host`、`Client`和`Server`三个层次：

- `Host`是运行在用户侧的宿主应用，例如`Claude Desktop`、`Cursor`、`Continue`这类工具。它负责创建并管理多个`Client`实例，同时掌管安全策略、用户授权以及与大模型的交互。对用户而言，`Host`就是他直接打交道的那个程序。

- `Client`由`Host`创建，每个`Client`与一个`MCP Server`保持`1:1`的有状态连接。它处理协议层面的握手、能力协商、消息路由，并在连接之间维持严格的安全隔离，一个`Client`看不到另一个`Client`连接的`Server`的任何内容。

- `Server`是工具的提供方，也是开发者最需要关心的角色。它以独立进程或远程服务的形式存在，对外暴露`Tool`、`Resource`、`Prompt`三类原语，专注于自己负责的那一块能力，不感知也无法访问完整的对话上下文。

这三者的关系可以类比为：`Host`是浏览器，`Client`是浏览器开的每一个标签页中的网络层，`Server`是每个标签页背后的`Web`服务。`Host`统一调度，`Client`维持连接，`Server`提供内容。一个`Host`可以同时持有多个`Client`，每个`Client`对应一个`Server`，这就是为什么`Cursor`可以同时接入天气、数据库、代码检索多个`MCP Server`并按需调用。

所有`MCP`消息都以`JSON-RPC 2.0`格式编码，这是协议的基础骨架。它定义了三类消息：

- 请求（`Request`）：`Client`向`Server`发起，携带`id`、`method`与`params`，期望得到响应。

- 响应（`Response`）：`Server`针对请求返回，携带相同的`id`，内容为`result`（成功）或`error`（失败）。

- 通知（`Notification`）：单向推送，不携带`id`，不期望回复，双方均可发出。例如`Server`工具列表发生变更时，会向`Client`发送`notifications/tools/list_changed`通知。

一条典型的工具调用请求如下所示：

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "tools/call",
  "params": {
    "name": "get_weather",
    "arguments": {
      "city": "北京",
      "date": "2026-05-08"
    }
  }
}
```

`Server`的正常响应结构如下所示：

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "result": {
    "content": [{
      "type": "text",
      "text": "北京当前天气：晴，气温22摄氏度，湿度35%，西北风3级"
    }],
    "isError": false
  }
}
```

工具业务执行失败时，应将`isError`置为`true`并在`content`中填入错误描述，而非依赖`JSON-RPC`层面的`error`字段。`JSON-RPC`的`error`仅用于表达协议层错误，如方法不存在、入参格式非法等，二者语义不应混用。
