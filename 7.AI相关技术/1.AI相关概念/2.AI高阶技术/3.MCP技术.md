回顾上一节描述的工具调用流程，平台在解析出大模型返回的工具名称与入参后，需要知道「去哪里调用、怎么调用」，也就是维护一份工具名称到调用地址、认证方式的配置映射，然后自行拼装`HTTP`请求发出去。这套做法的问题在于：每接入一个新工具，平台侧都要单独适配，编写对应的调用逻辑、处理各自的认证方式与数据格式，扩展成本很高。

`MCP`解决的正是这个问题。它定义了一套标准协议，规定工具服务端（`MCP Server`）必须以统一的格式暴露自己的能力，平台侧（`MCP Client`）只需实现一套通用的客户端逻辑，便能与任意符合协议的工具服务通信，无需再为每个工具单独适配。

在生活中类比一下就是，没有`MCP`之前，平台对接每个工具，就像给不同品牌的设备分别配一根专用充电线；有了`MCP`，相当于统一换成了`USB-C`接口，工具只要支持这个标准，平台插上就能用。

需要澄清一个常见误区：「平台将工具列表提供给大模型」以及「大模型将工具调用意图返回给平台」这两个环节的`JSON`结构，并不属于`MCP`的规范范畴。这份`JSON`结构属于大模型`API`层面的协议，由各模型厂商自行定义，不同厂商的格式存在差异，是模型接口层面的问题。`MCP`规范的仅是「平台如何与工具交互」这一环节。

`MCP`在通信格式上基于`JSON-RPC 2.0`，所有交互均遵循统一的请求/响应结构，协议主要规范了能力发现与工具调用两个环节。

平台连接到`MCP Server`后，首先调用`tools/list`接口，询问「你有哪些工具」。`MCP Server`以统一格式返回工具列表：

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "tools": [
      {
        "name": "get_weather",
        "description": "查询指定城市的实时天气信息",
        "inputSchema": {
          "type": "object",
          "properties": {
            "city": { "type": "string", "description": "城市名称" },
            "date": { "type": "string", "description": "查询日期，格式YYYY-MM-DD" }
          },
          "required": ["city", "date"]
        }
      }
    ]
  }
}
```

平台获取工具列表后，`MCP Client`将其转换为大模型`API`所要求的格式，随请求一并发送给大模型，整个过程对`MCP Server`透明。

大模型返回工具调用意图后，平台解析出工具名称与入参，再通过`MCP`协议向`MCP Server`发起调用，使用`tools/call`接口：

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "tools/call",
  "params": {
    "name": "get_weather",
    "arguments": {
      "city": "北京",
      "date": "2026-04-18"
    }
  }
}
```

`MCP Server`处理完成后，同样以统一格式返回执行结果：

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "{\"city\":\"北京\",\"condition\":\"晴\",\"temperature\":22}"
      }
    ],
    "isError": false
  }
}
```

上一节中，平台是自己维护配置、自己拼`HTTP`请求，每个工具的请求路径、认证方式、入参格式各不相同。引入`MCP`后，平台只需与`MCP Server`之间跑统一的`JSON-RPC`协议，工具内部怎么实现、去调哪个下游接口，平台完全不用关心。`MCP`把「工具名称到调用地址的配置映射」以及「拼装`HTTP`请求的适配逻辑」，全部下沉到了`MCP Server`内部，平台侧只保留一套通用的调用逻辑。
