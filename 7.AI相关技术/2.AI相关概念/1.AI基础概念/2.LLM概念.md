`LLM`的全称是`Large Language Model`，中文通常翻译为“大语言模型”，简称大模型。

目前几乎所有主流大模型都基于`Transformer`模型。`Transformer`架构的核心在于`Self-Attention`（自注意力机制）。

在传统的循环神经网络（`RNN`）中，模型按顺序处理句子，必须先处理前一个词，才能继续处理下一个词。这种机制容易导致模型“遗忘”句首信息。而`Self-Attention`具有全局视野，在处理每个`Token`时都会同时关注句子中的所有其他词，并为其分配不同权重。

例如句子：“那个奖杯放不进手提箱，因为它太大了”，当模型处理“它”时，会为“奖杯”分配较高权重。这是因为在常识中，“大”通常导致“放不进”，而“放不进”的主体通常是被放置的物体，也就是奖杯。

如果将句子改为：“那个奖杯放不进手提箱，因为它太小了”，此时模型在处理“它”时，就会将权重转移到“手提箱”上。虽然同样是“放不进”，但原因变为“小”。在常识中，容器（手提箱）过小才会导致空间不足。

`Self-Attention`的计算本质是：每个`Token`会被映射为三组向量：`Query`、`Key`和`Value`。通过计算`Q`与`K`的点积得到权重，再将该权重作用于`V`，从而实现对长距离依赖关系的建模。

大模型通常需要先经历一个规模巨大的训练阶段，这一过程称为预训练（`Pretraining`）。其核心目标是使模型在海量文本数据上学习语言的统计规律以及上下文依赖关系。预训练采用自监督学习（`Self-Supervised Learning`）范式，模型在持续最小化预测误差的过程中，将语言知识逐步编码进参数，从而具备基础的生成能力。

预训练的核心前提是先确定模型参数量（即日常所说的“多少`B`”，`B=Billion`，十亿）。参数量的大小并非随意设定，而是由硬件能力决定：硬件的显存总量、`GPU`数量以及整体集群算力，直接决定了模型规模的上限。例如，消费级显卡通常只能支撑`7B`、`13B`等中小规模模型；而千亿级（`100B`以上）模型，则需要上万张专业`GPU`（如`A100`、`H100`）组成的大规模计算集群才能完成训练。

在确定参数规模之后，需要配套相应规模的预训练语料。行业普遍遵循`Scaling Law`（缩放定律），即模型参数量越大，所需训练`Token`数量也越多。例如，`7B`模型通常需要万亿级`Token`语料，而`70B`模型则往往需要十万亿级`Token`语料。

如果数据规模与模型参数规模不匹配，模型可能无法有效学习，从而出现过拟合或欠拟合问题。参数多而数据少容易过拟合，模型仅记住训练数据，泛化能力差；参数少而数据多容易欠拟合，模型容量不足，无法充分学习知识。

从概率建模的角度看，模型实际上是在拟合一个条件概率分布：在给定已有`Token`序列的条件下，预测下一个`Token`出现的概率。

训练数据通常来源于互联网文本、书籍以及代码等多种语料，并通过`Tokenizer`统一转换为`TokenID`序列，作为模型训练的输入。但仅依赖预训练阶段，模型生成内容的质量通常仍然较低，因此需要通过微调（`Fine-tuning`）进一步对齐人类的指令与偏好。完成这一系列训练之后，模型才会进入最终的推理（`Inference`）阶段。

在上一节中提到，大模型不会直接处理文本，而是先通过`Tokenizer`将文本拆分为多个`Token`，再根据预训练分词词表映射为`TokenID`列表。随后，每个`TokenID`会经过`Embedding`层（嵌入层）映射为高维向量，并叠加位置编码，最终形成同时包含语义信息与顺序信息的数值矩阵，作为模型计算的输入。

其中，高维向量可以理解为由多个浮点数组成的数组，长度通常为几百到几千。向量中的每一个维度并不直接对应某个具体含义，而是模型在训练过程中自动学习到的特征表示。整体来看，这些维度共同刻画了`Token`在语义空间中的位置，使得语义相近的`Token`在向量空间中的距离更接近，而语义差异较大的`Token`则相距更远。

大模型并不进行思考，而是执行一种基于`Token`的序列预测过程，本质上类似于单`Token`的接龙。在模型经过多层计算并处理输入向量后，会生成一个融合了上下文信息的高维向量，用于表达对“下一个词”的语义预测。

为了将这一高维向量转换为人类可读的文本，模型需要将其映射回预训练分词词表，得到在当前语境下各个词出现的概率分布。随后依据既定的采样策略，从中选取一个合适的`TokenID`，最终再通过`Tokenizer`查表还原为对应的自然语言文本。

我们可以先不考虑`TokenID`与高维向量之间的映射过程，只聚焦单个`Token`逐步生成的本质。

例如输入“蒙多想去哪就去哪”，`Tokenizer`会将其转换为`TokenID`序列`[75, 61, 18, 22, 12, 22]`。该序列输入模型后，模型基于已有上下文预测下一个`Token`为“是”，对应的`TokenID`为`32`。此时序列更新为`[75, 61, 18, 22, 12, 22, 32]`。

接着，这个新序列会再次输入模型，继续预测下一个`Token`，如此循环。既然是“接龙”，就必须有一个停止条件。在整个词表中，通常会包含一个特殊的“隐形`Token`”，称为`[EOS]`（`End Of Sequence`，终止符）。当模型在某一步计算中，发现概率最高，或者通过采样选中的`TokenID`正好对应这个`[EOS]`时，就表示当前序列已经完整，生成过程应当结束。

在`Decoding`（解码）阶段，为提升用户体验，当前主流`LLM`通常采用`Stream`（流式传输）机制，即模型每生成一个`Token`，就会立即完成解码并返回给用户，因此在实际使用大模型平台时，会观察到逐步流式输出的效果。

这种`Token`接龙的过程在技术上称为自回归生成（`Autoregressive Generation`）。从工程角度看，从接收指令到完成接龙的整个运行过程，被统称为“推理”（`Inference`）。

大模型会通过`KV Cache`技术，将已计算过的`Token`特征缓存起来；在每一轮生成时，仅需对最新的一个`Token`进行计算，从而避免大规模的重复算力消耗。在`API`计费层面，费用是基于输入总量与生成总量进行统计的，并不会因为循环调用次数而重复计费。

用户输入到大模型并得到输出的整体流程，其`PlantUML`代码如下所示：

```scss
@startuml
actor 用户
participant "Tokenizer" as Tokenizer
participant "TokenID序列" as Tokens
participant "大语言模型" as LLM
participant "概率分布+采样策略" as Sampling

用户 -> Tokenizer : 输入文本
Tokenizer -> Tokens : 编码，转换为TokenID序列

loop 逐Token生成
    Tokens -> LLM : 输入当前序列
    LLM -> Sampling : 计算概率并采样下一个Token
    Sampling --> Tokens : 返回TokenID，追加到序列

    Tokens -> Tokenizer : 单TokenID解码
    Tokenizer --> 用户 : 流式输出对应文本片段

    alt 已生成EOS
        Tokenizer --> 用户 : 代表已输出完整结果
    else 未生成EOS
        Tokens -> LLM : 输入当前序列，重复上述步骤
    end
end
@enduml
```
