接下来的内容以概念居多，有不太明白的地方先不用细究，后面都会详细解释。

在Kubernetes（K8s）中，所有操作都以资源为抽象单位，用户通过对这些资源的操作来有效地管理整个集群。

Kubernetes的核心是构建在集群系统之上，用户可以轻松地在集群中部署和运行各种服务。服务的部署涉及在K8s集群中启动容器，将特定程序运行在这些容器中。

Kubernetes并非直接管理容器，而是以Pod为最小管理单元。Pod是一组密切关联的容器集合，它们共享网络和存储资源。通常情况下，Kubernetes通过PodController来管理Pod，确保它们始终处于用户期望的状态。

为了提供稳定的网络标识和访问方式，K8s引入了Service资源。Service充当一个抽象层，为一组Pod提供了统一的访问入口。通过ClusterIP、NodePort、LoadBalancer等方式，Service能够将流量引导到正在运行的Pod。

除了网络访问之外，如果Pod中运行的应用程序需要持久化存储，Kubernetes提供了各种存储系统的抽象，其中包括Persistent Volume Claims（PVC）、ConfigMap等。

这种架构使得Kubernetes成为一个强大而灵活的容器编排平台，通过资源抽象和高度自动化的管理，简化了复杂的应用程序部署和维护过程。

学习kubernetes的核心，就是学习如何对集群上的**Pod、Pod控制器、Service、存储**等各种资源进行操作。

我们一般如何操作这些资源？这就引出了以下三种资源管理方式：

1. 命令式对象管理：直接使用命令去操作kubernetes资源，例如：

```bash
kubectl run nginx-pod --image=nginx:1.17.1 --port=80
```

2. 命令式对象配置：通过命令配置和配置文件去操作kubernetes资源，例如：

```bash
kubectl create/patch -f nginx-pod.yaml
```

3. 声明式对象配置：通过apply命令和配置文件去操作kubernetes资源，例如：

```bash
kubectl apply -f nginx-pod.yaml
```

命令式对象管理是通过kubectl等命令行工具直接操作Kubernetes对象的状态。

命令式对象配置和声明式对象配置都是操作`yaml`文件，但命令式对象配置是直接操作 Kubernetes 对象，而声明式对象配置则是描述 Kubernetes 对象的期望状态，让 Kubernetes 自己去实现这个状态。

| 类型           | 适用环境 | 优点                 | 缺点                                       |
| -------------- | -------- | -------------------- | ------------------------------------------ |
| 命令式对象管理 | 测试     | 操作简单，直观       | 仅能处理活动对象，难以审计和跟踪           |
| 命令式对象配置 | 开发     | 易审计、可跟踪，灵活 | 随项目规模增大，配置文件数量增多，操作繁琐 |
| 声明式对象配置 | 开发     | 支持目录操作，易扩展 | 难以在意外情况下进行调试，配置复杂度较高   |

这三种配置会在后面章节进行更加详细的讲解。
